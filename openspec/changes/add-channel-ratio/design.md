## Context

### 背景
New API 是一个 AI 网关系统，需要对不同渠道（AI 服务提供商）的请求进行计费。当前计费公式为：

```
费用 = Token数 × 模型倍率 × 用户分组倍率
```

运营需要在不修改上游模型倍率的情况下，对不同渠道设置不同的价格策略。

### 约束
- 模型倍率由上游提供，不可随意修改
- 渠道倍率需要对用户隐藏（后台配置）
- 需要与现有的用户分组倍率机制兼容
- 重试时可能切换渠道，需要处理费用差异

### 利益相关方
- 运营团队：需要灵活的定价策略
- 用户：看到统一的定价，不感知渠道差异
- 开发团队：需要维护计费逻辑的正确性

## Goals / Non-Goals

### Goals
1. 为渠道添加可配置的倍率字段
2. 扩展计费公式，将渠道倍率纳入计算
3. 对用户隐藏渠道倍率细节
4. 正确处理重试时渠道切换导致的费用变化
5. 保持向后兼容（默认倍率 1.0）

### Non-Goals
1. 不实现渠道分组/标签批量管理（可作为后续优化）
2. 不修改现有的用户分组倍率逻辑
3. 不对外暴露渠道倍率 API（仅管理员可见）

## Decisions

### Decision 1: 渠道倍率存储方式

**选择**: 在 Channel 表直接添加 `ratio` 字段

**原因**:
- 实现简单，一个字段搞定
- 每个渠道独立配置，灵活度高
- 与现有 Channel 模型一致，无需额外表

**替代方案**:
- ❌ 渠道分组+分组倍率：增加额外的配置层级，复杂度高
- ❌ 渠道标签+标签倍率：需要额外的映射表，初期不必要

### Decision 2: 计费公式

**选择**: `费用 = Token × 模型倍率 × 用户分组倍率 × 渠道倍率`

**原因**:
- 三个倍率相乘，简单直观
- 与现有公式自然扩展
- 各倍率独立，职责清晰

### Decision 3: 重试时费用处理

**选择**: 重试前退回预扣费，按新渠道重新计算

**流程**:
```
1. 首次请求：按渠道A预扣费 Q1
2. 渠道A失败，需要重试到渠道B
3. 退回预扣费 Q1
4. 按渠道B重新计算预扣费 Q2
5. 扣除 Q2
6. 请求完成后，按实际消耗结算
```

**原因**:
- 保证计费精确
- 用户最终只按实际使用的渠道付费
- 避免多扣或少扣

### Decision 4: 用户可见性

**选择**: 渠道倍率完全对用户隐藏

**实现**:
- 定价页面：只展示模型基础价格 × 用户分组倍率
- 账单详情：只展示模型名、Token数、总扣费，不展示渠道信息
- 管理后台：完整展示所有信息（含渠道倍率）

**原因**:
- 用户关心的是"用了多少钱"，不是"用了哪个渠道"
- 简化用户理解成本
- 渠道选择是平台内部决策

## Risks / Trade-offs

### Risk 1: 用户困惑于费用波动
**描述**: 同一个模型不同请求可能费用不同（因为系统选择了不同渠道）
**影响**: 中等
**缓解措施**:
- 账单不显示单价，只显示总扣费
- 如有用户咨询，客服可解释为"服务成本波动"

### Risk 2: 配置错误导致亏损
**描述**: 管理员配置渠道倍率过低，导致收费低于成本
**影响**: 高
**缓解措施**:
- 倍率字段添加范围约束（0.1 ~ 100）
- 管理后台添加成本分析报表
- 配置变更时提示确认

### Risk 3: 重试逻辑复杂化
**描述**: 重试时需要处理费用退还和重新扣费
**影响**: 中等
**缓解措施**:
- 复用现有的 ReturnPreConsumedQuota 机制
- 添加完整的日志记录便于排查
- 单元测试覆盖重试场景

## Migration Plan

### Phase 1: 数据库迁移
1. 为 `channels` 表添加 `ratio` 列，默认值 1.0
2. 现有渠道自动使用默认值，无需手动迁移

### Phase 2: 后端实现
1. 更新 Channel 模型
2. 修改计费逻辑
3. 实现重试费用处理

### Phase 3: 前端实现
1. 渠道编辑页面添加倍率字段
2. 确保定价页面和账单页面不展示渠道倍率

### Rollback Plan
1. 数据库：保留 `ratio` 列，不删除
2. 代码：回滚到不使用渠道倍率的版本
3. 效果：所有渠道恢复按倍率 1.0 计费
