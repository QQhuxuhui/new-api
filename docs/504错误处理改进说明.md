# 504 错误处理改进说明

## 改进概述

**改进日期**: 2025-11-20
**改进类型**: 保守改进
**影响范围**: 超时错误（504/524）的健康检查记录

## 问题背景

之前当上游 API 返回 504 Gateway Timeout 错误时：
- ❌ 不记录到健康检查系统
- ❌ 频繁 504 的通道不会被自动暂停
- ❌ 无法利用健康检查机制自动恢复

**日志示例**:
```
[ERR] channel error (channel #4, status code: 504): bad response status code 504
[INFO] record error log: userId=4, channelId=4, content=bad response status code 504
[ERR] relay error: bad response status code 504
```

## 改进方案

### 保守策略（已实施）

1. **504/524 不触发立即重试**（保持原有行为）
   - 避免重试导致雪崩效应
   - 由客户端决定是否重试

2. **504/524 记录到健康检查系统**（新增）
   - 累积到滑动窗口统计中
   - 如果失败率 >30%，自动暂停通道
   - 利用指数退避机制恢复

3. **自动恢复机制**
   - 第 1 次暂停：5 分钟
   - 第 2 次暂停：10 分钟
   - 第 3 次暂停：20 分钟
   - 最大暂停：60 分钟

## 代码改动

**文件**: `controller/relay.go:196-203`

```go
// 改动前
if service.ShouldTriggerChannelFailover(newAPIError.StatusCode, newAPIError.Error()) {
    service.RecordChannelFailure(channel.Id, newAPIError.StatusCode, newAPIError.Error())
}

// 改动后
if service.ShouldTriggerChannelFailover(newAPIError.StatusCode, newAPIError.Error()) ||
    newAPIError.StatusCode == 504 || newAPIError.StatusCode == 524 {
    service.RecordChannelFailure(channel.Id, newAPIError.StatusCode, newAPIError.Error())
}
```

## 预期效果

### 场景 1: 偶发 504
- 通道偶尔返回 504（失败率 <30%）
- **行为**: 记录到统计中，但不暂停通道
- **结果**: 不影响正常服务

### 场景 2: 频繁 504
- 通道持续返回 504（失败率 >30%）
- **行为**: 自动暂停通道 5 分钟
- **结果**: 流量切换到其他健康通道

### 场景 3: 通道恢复
- 暂停期满后，通道恢复正常
- **行为**: 成功请求重置暂停计数
- **结果**: 通道恢复正常使用

## 健康检查参数

| 参数 | 值 | 说明 |
|-----|----|----|
| 滑动窗口 | 60 秒 | 统计周期 |
| 失败率阈值 | 30% | 触发暂停的失败率 |
| 低流量阈值 | 50% | 低流量通道的失败率阈值 |
| 最小样本数 | 5 | 最少请求数才开始评估 |
| 基础暂停时间 | 5 分钟 | 第一次暂停时长 |
| 最大暂停时间 | 60 分钟 | 暂停时长上限 |
