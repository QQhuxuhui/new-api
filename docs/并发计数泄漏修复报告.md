# 并发计数泄漏问题排查与修复报告

## 问题描述

线上 `claude_主力` 渠道显示有 3 个并发一直下不去，怀疑是数据不一致导致的通道一直显示并发，但实际没有使用。

## 排查过程

### 1. 连接 Redis 诊断

创建了诊断脚本 `scripts/diagnose_concurrency.go`，连接远程 Redis (sparkcode.top:6379) 进行检查。

**发现问题：**
- 2 个 Redis key 存在泄漏的并发计数
- 每个 key 的并发数都是 3
- 所属渠道类型：Anthropic (Claude) (channelType=14)
- TTL 剩余约 1 小时

```
Key: channel:key:808f45ff7188c3c939ea5529846dd0579059b296e449b364d1c5f8985d369a8f:type:14:concurrent
  当前并发数: 3
  TTL: 58m9s

Key: channel:key:25dc7f88aa617386a4c844a6de4927cb078af2bda6028b7e134917b54f960bec:type:14:concurrent
  当前并发数: 3
  TTL: 59m49s
```

### 2. 根本原因分析

通过代码审查，发现了以下潜在问题：

#### 问题 1: defer 函数缺少空字符串检查
**位置：** `middleware/distributor.go:289`

```go
// 修复前
defer func() {
    if concurrencyKey, exists := c.Get("concurrency_key"); exists {
        if key, ok := concurrencyKey.(string); ok {  // ❌ 没有检查 key != ""
            channelType := common.GetContextKeyInt(c, constant.ContextKeyChannelType)
            service.DecrementConcurrency(key, channelType)
        }
    }
}()
```

当重试发生时，旧的 `concurrency_key` 被设置为空字符串（line 508），但 defer 函数只检查了类型转换是否成功，没有检查是否为空字符串。

#### 问题 2: 服务崩溃或重启导致 defer 未执行
如果在请求处理过程中：
- 服务被强制杀死（SIGKILL）
- 发生严重错误导致进程崩溃
- Redis 操作超时失败

defer 函数可能不会执行或执行失败，导致并发计数永久泄漏。

## 修复方案

### 修复 1: 添加空字符串检查

**文件：** `middleware/distributor.go:289`

```go
// 修复后
defer func() {
    if concurrencyKey, exists := c.Get("concurrency_key"); exists {
        if key, ok := concurrencyKey.(string); ok && key != "" {  // ✓ 添加 key != "" 检查
            channelType := common.GetContextKeyInt(c, constant.ContextKeyChannelType)
            service.DecrementConcurrency(key, channelType)
        }
    }
}()
```

### 修复 2: 添加定期清理机制

**新增文件：** `service/concurrency_cleanup.go`

创建了一个后台清理任务，每 10 分钟扫描一次 Redis 中的并发计数 key：
- 如果 TTL < 5 分钟，认为是泄漏的计数（正常应该立即删除）
- 自动清理这些可疑的 key

**启动位置：** `main.go:96`

```go
// 启动并发计数清理任务（防止因服务崩溃导致的并发泄漏）
service.StartConcurrencyCleanupTask()
```

### 修复 3: 提供手动清理工具

**新增文件：** `scripts/cleanup_concurrency.go`

提供了手动清理脚本，可以一键清理所有泄漏的并发计数：

```bash
REDIS_CONN_STRING=redis://sparkcode.top:6379 go run scripts/cleanup_concurrency.go
```
