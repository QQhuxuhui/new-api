🔄 3. 完整请求处理流程

客户端发起请求

支持的协议：HTTP/HTTPS, WebSocket

请求格式：

  • OpenAI 格式：/v1/chat/completions
  • Claude 格式：/v1/messages
  • Gemini 格式：/v1beta/models/xxx:generateContent
  • 实时 API：/v1/realtime
⬇️

路由层 - 路由匹配

文件：router/relay-router.go

功能：

  • 根据请求路径匹配对应的处理器
  • 区分不同的 API 格式（OpenAI/Claude/Gemini）
  • 应用全局中间件链
⬇️

中间件 - Token 认证 (TokenAuth)

文件：middleware/auth.go

认证流程：

  1. 多协议提取 Token：
    • Bearer Token（HTTP Authorization header）
    • WebSocket（Sec-WebSocket-Protocol）
    • Anthropic（x-api-key header）
    • Gemini（query parameter 或 x-goog-api-key）
  2. Token 验证：
    • 检查 Token 状态（enabled/exhausted/expired）
    • 验证过期时间
    • 验证剩余配额
  3. IP 白名单检查（如果配置）
  4. 客户端限制检查（新特性）：
    • 提取 User-Agent
    • 支持精确匹配、通配符、正则表达式
    • 防止 API Key 滥用
  5. 粘性会话配置（新特性）：
    • 如果 Token 启用粘性会话
    • 设置粘性会话标志和 TTL
  6. 用户权限验证
  7. 设置请求上下文
⬇️

中间件 - 模型限流 (ModelRequestRateLimit)

文件：middleware/model-rate-limit.go

功能：

  • 基于模型的请求速率限制
  • 使用 Token Bucket 或 Leaky Bucket 算法
  • 防止单个用户过度消耗资源
⬇️

中间件 - 渠道分配 (Distribute)

文件：middleware/distributor.go

分配逻辑：

  1. 解析请求获取模型名称
  2. Token 模型权限验证（如果启用）
  3. 渠道选择策略：

    🔄 粘性会话模式（Sticky Session）：

    • 获取会话用户 ID（userId/sessionId）
    • 查询是否已绑定渠道：Redis: sticky_session:{userId}:{model}:{group}
    • 如果存在绑定：
      • 验证渠道健康状态
      • 健康 ✅ → 使用绑定渠道，更新 TTL
      • 失效 ❌ → 解绑，重新选择
    • 如果不存在绑定：
      • 调用渠道选择算法
      • 绑定新渠道到会话
      • 设置 TTL（默认 1 小时）

    🎲 普通模式（Random Selection）：

    • 调用 CacheGetRandomSatisfiedChannel(group, model, 0)
    • 使用加权随机算法选择渠道
  4. 获取渠道 Key（支持多 Key 管理）：
    • Random 模式：随机选择可用 Key
    • Polling 模式：轮询选择可用 Key
    • 使用渠道级别锁保证并发安全
  5. 设置渠道上下文到请求
⬇️

控制器 - 转发处理 (Relay)

文件：controller/relay.go

处理流程：

  1. 请求验证和解析
  2. 敏感词检测（如果启用）
  3. Token 计数（提示词 token 数）
  4. 价格计算（基于模型和 token 数）
  5. 预扣费（从用户配额中预扣）
  6. 重试循环（最多 RetryTimes 次）：
    • 首次：使用 Distribute 选择的渠道
    • 重试：重新选择渠道（降级到低优先级）
    • 调用适配器处理请求
    • 成功 → 返回响应
    • 失败 → 错误处理，判断是否重试
  7. 配额结算（成功）或 退款（失败）
⬇️

转发层 - 适配器处理

文件：relay/relay_adaptor.go

适配器流程：

  1. 获取适配器： adaptor := GetAdaptor(channelType)
  2. 初始化： adaptor.Init(relayInfo)
  3. 请求转换：
    • 模型映射（model_mapping）
    • 参数覆盖（param_override）
    • 格式转换（OpenAI ↔ Claude ↔ Gemini）
  4. 构建请求 URL（支持自定义 BaseURL）
  5. 设置请求头：
    • 认证信息（API Key / Organization）
    • 请求头覆盖（header_override）
  6. 发送请求到上游 API
  7. 处理响应（流式/非流式）
  8. 解析 usage（token 消耗统计）

📡 5. 渠道配置详解

💡 提示： 渠道配置在管理后台的 "渠道管理" 页面进行设置，对应前端组件：web/src/pages/Channel/

📋 核心配置字段

配置项 类型 说明 作用
Type 整数 渠道类型 ID 标识 AI 服务商类型：
1=OpenAI, 8=Claude, 15=Gemini, 等
Key 字符串 API Key（支持多 Key） 认证凭证，支持两种格式：
• 换行分隔：sk-xxx\nsk-yyy
• JSON 数组：["sk-xxx", "sk-yyy"]
Status 整数 渠道状态 1=启用, 2=手动禁用, 3=自动禁用
Name 字符串 渠道名称 便于识别和管理
Weight 整数 权重（0-999） 负载均衡权重，数字越大被选中概率越高。
算法：加权随机算法
如果所有权重为 0，每个渠道视为权重 100
Priority 整数 优先级 优先级分层，数字越大优先级越高。
重试逻辑：失败后降级到低优先级渠道
BaseURL 字符串 自定义 API 端点 覆盖默认的上游 API 地址。
用途：支持中转节点、自建服务
Models 字符串 支持的模型列表 逗号分隔的模型名称，例如：
gpt-4,gpt-3.5-turbo,claude-3-opus
Group 字符串 渠道分组 逗号分隔的分组名称，例如：
default,premium,enterprise
用途：不同用户组使用不同渠道集合
ModelMapping JSON 模型映射 将请求的模型名映射到上游模型：
{"gpt-4": "gpt-4-turbo"}
用途：统一模型名称、兼容性处理
ParamOverride JSON 参数覆盖 强制覆盖请求参数：
{"temperature": 0.7, "max_tokens": 2000}
用途：限制用户参数、统一配置
HeaderOverride JSON 请求头覆盖 添加或覆盖 HTTP 请求头：
{"X-Custom-Header": "value"}
用途：中转节点认证、自定义标识
AutoBan 整数 自动禁用开关 启用后，渠道出现特定错误会自动禁用：
• 401 Unauthorized
• 429 Too Many Requests（配额耗尽）
• 其他致命错误
Tag 字符串 渠道标签 用于批量管理渠道，例如：
official, backup, testing

🔧 多 Key 管理配置

多 Key 管理模式

配置位置：渠道编辑 → 多 Key 设置

配置项 说明
MultiKeyMode Random（随机）：随机选择可用 Key
Polling（轮询）：顺序轮询可用 Key
推荐：高并发场景使用 Random，顺序场景使用 Polling
MultiKeyStatusList 每个 Key 的状态映射：
{0: 1, 1: 1, 2: 3}
（索引 → 状态，1=启用, 2=手动禁用, 3=自动禁用）
MultiKeyDisabledReason 禁用原因记录：
{2: "401 Unauthorized"}
MultiKeyPollingIndex 轮询模式下的当前索引位置
⚠️ 并发安全： 多 Key 管理使用渠道级别的锁（channelPollingLocks）保证并发安全，防止同时读写状态列表。

⚖️ 负载均衡算法

加权随机算法

实现文件：model/channel_cache.go::GetRandomSatisfiedChannel()

// 算法步骤： 1. 从缓存获取可用渠道列表 channels := group2model2channels[group][model] 2. 按优先级分组 uniquePriorities := getUniquePriorities(channels) sortedPriorities := sortDescending(uniquePriorities) 3. 根据 retry 次数选择优先级（重试时降级） targetPriority := sortedPriorities[min(retry, len(sortedPriorities)-1)] 4. 筛选目标优先级的渠道 targetChannels := filterByPriority(channels, targetPriority) 5. 权重归一化 - 如果所有权重为 0: 每个渠道有效权重 = 100 - 如果平均权重 < 10: 应用平滑因子 100 6. 加权随机选择 sumWeight := calculateTotalWeight(targetChannels) randomWeight := rand.Intn(sumWeight) for _, channel := range targetChannels { randomWeight -= channel.Weight if randomWeight < 0 { return channel // 选中 } }
✅ 优势：
  • 优先级分层：主渠道优先，失败后自动降级到备份渠道
  • 权重分配：灵活控制流量分布
  • 平滑处理：避免权重配置错误导致选择失败

🆕 新特性详解

🔄 粘性会话（Sticky Session）

实现文件：
service/session.go
middleware/distributor.go
工作原理：
1. 首次请求：选择渠道并绑定
2. 存储：Redis: sticky_session:{userId}:{model}:{group}
3. 后续请求：优先使用绑定渠道
4. 健康检查：渠道失效时自动重绑
5. TTL 过期：自动解绑，下次重新选择
优势：
• 减少渠道切换
• 提升用户体验
• 避免上下文丢失
• 支持会话状态保持

🛡️ 客户端限制（Client Restriction）

实现文件：
middleware/auth.go::checkClientRestriction()
匹配逻辑：
1. 提取请求的 User-Agent
2. 遍历允许的客户端列表
3. 应用匹配规则：
  • 精确匹配
  • 通配符匹配（glob）
  • 正则表达式匹配
4. 任意一条匹配成功 → 通过
5. 全部不匹配 → 拒绝（403）
应用场景：
• 限制只允许特定应用访问
• 防止 API Key 泄露后被滥用
• 实现客户端级别的访问控制
• 区分不同客户端的使用权限

📝 客户端限制配置示例

# AllowedClients 配置示例（换行分隔） # 1. 精确匹配（完全一致才通过） Claude-Code-CLI/1.0.0 PostmanRuntime/7.32.3 # 2. 通配符匹配（* 匹配任意字符） VSCode/* Claude-Code-CLI/* curl/* # 3. 正则表达式匹配（以 regex: 开头） regex:^Postman/.* regex:^curl/[0-9]+\\..* regex:^Mozilla/5.0.*Chrome.* # 4. 组合使用 Claude-Code-CLI/* VSCode/* regex:^Postman/.*
💡 提示：
  • 每行一个规则
  • 精确匹配效率最高，优先使用
  • 通配符匹配简单场景
  • 正则表达式匹配复杂场景
  • 支持注释（以 # 开头的行会被忽略）

4️⃣ 自动重试机制

实现文件：controller/relay.go::shouldRetry()

// 重试策略 for retry := 0; retry <= RetryTimes; retry++ { if retry > 0 { // 重新选择渠道（降级到低优先级） channel = CacheGetRandomSatisfiedChannel(group, model, retry) } err = relayHandler(c, relayInfo) if err == nil { return // 成功 } if !shouldRetry(c, err, RetryTimes-retry) { break // 不应该重试 } } // 重试条件：429, 307, 5xx (除超时) // 不重试：400, 408, 504, 524, 2xx, Skip Retry Error
✅ 优势： 提高成功率、自动降级、智能判断

💡 8. 配置最佳实践

🎯 渠道配置建议

1. 优先级配置

场景 优先级配置 说明
官方渠道 优先级 = 100 主要流量，优先使用
备用渠道 优先级 = 50 官方失败时使用
测试渠道 优先级 = 10 最低优先级，仅做备份

2. 权重配置

场景 权重配置 流量分布
主渠道 A 权重 = 70 70% 流量

3. 多 Key 配置

  • 高并发场景：使用 Random 模式，分散请求压力
  • 顺序访问：使用 Polling 模式，均匀分配
  • Key 数量：建议 3-5 个，过多管理复杂
  • 监控：定期检查单个 Key 状态，及时处理禁用

🔑 Token 配置建议

1. 粘性会话配置

场景 是否启用 TTL 设置 说明
对话类应用 ✅ 启用 3600 秒（1 小时） 保持上下文，减少切换
单次调用 ❌ 不启用 - 无需会话状态
长时间会话 ✅ 启用 7200 秒（2 小时） 延长绑定时间

2. 客户端限制配置

  • 生产环境：启用客户端限制，防止 Key 泄露
  • 内部应用：精确匹配客户端名称
  • 第三方集成：使用通配符匹配版本号
  • 调试环境：可暂时禁用，便于测试
⚠️ 安全提醒：
  • 生产 Token 必须启用客户端限制或 IP 白名单
  • 定期轮换 Token，降低泄露风险
  • 监控异常 User-Agent，及时发现滥用

3. 配额管理建议

  • 普通用户：设置合理配额，防止滥用
  • 付费用户：提高配额或启用无限配额
  • 内部 Token：无限配额，不过期
  • 测试 Token：小额配额，短期过期
