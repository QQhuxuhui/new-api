# New API 容灾和负载均衡配置指南

## 📋 文档概述

本文档详细讲解如何通过配置实现高可用性和自动故障转移,确保服务稳定运行。

### 学习目标
- ✅ 理解容灾和负载均衡的原理
- ✅ 掌握权重、优先级的配置方法
- ✅ 学会设计多层容灾方案
- ✅ 能够实现自动故障切换

## 🎯 一、核心概念理解

### 1.1 什么是负载均衡?

**负载均衡** = 将请求分散到多个渠道,避免单个渠道过载。

**形象比喻**:

```
没有负载均衡:
所有客人 → 1号收银台 (排长队,很慢)
其他收银台 → 空闲

有负载均衡:
客人智能分流:
    60%客人 → 1号收银台
    30%客人 → 2号收银台
    10%客人 → 3号收银台
所有收银台合理利用
```

**工作原理**:

```
用户请求 → New API

New API根据规则选择渠道:
┌────────────────────────────┐
│  渠道A (权重20) → 50%流量  │
│  渠道B (权重15) → 37.5%流量 │
│  渠道C (权重5)  → 12.5%流量 │
└────────────────────────────┘
    ↓
上游API站点
```

### 1.2 什么是容灾?

**容灾** = 当部分渠道失败时,自动切换到可用渠道,保证服务不中断。

**场景演示**:

```
正常情况:
请求 → 渠道A → 成功 ✅

故障情况 (没有容灾):
请求 → 渠道A → 失败 ❌
用户报错,服务中断

故障情况 (有容灾):
请求 → 渠道A → 失败 ❌
      → 自动重试渠道B → 成功 ✅
用户无感知,服务正常
```

### 1.3 负载均衡 vs 容灾

| 维度 | 负载均衡 | 容灾 |
|------|---------|------|
| **目的** | 分散流量,提高性能 | 保证可用性,防止中断 |
| **工作时机** | 正常运行时 | 渠道故障时 |
| **机制** | 权重分配 | 失败重试、自动切换 |
| **配置方式** | 设置权重、优先级 | 配置重试次数、备用渠道 |

**两者配合使用**:

```
平时: 负载均衡分散流量到多个渠道
故障时: 自动切换到可用渠道(容灾)
结果: 既提高性能,又保证稳定性
```

## 🔧 二、负载均衡配置详解

### 2.1 权重分配机制

**权重的作用**: 决定每个渠道分配到的流量比例。

**计算公式**:

```
某渠道流量占比 = 该渠道权重 / 所有渠道总权重

示例计算:
渠道A: 权重 30
渠道B: 权重 20
渠道C: 权重 10
渠道D: 权重 5
总权重: 65

流量分配:
渠道A: 30/65 ≈ 46%
渠道B: 20/65 ≈ 31%
渠道C: 10/65 ≈ 15%
渠道D: 5/65 ≈ 8%
```

**权重设置原则**:

| 渠道质量 | 权重建议 | 流量占比 | 使用场景 |
|---------|---------|---------|---------|
| 🌟 顶级优质 | 18-20 | 30-40% | 主力渠道,承担核心流量 |
| ⭐ 优质 | 12-15 | 20-30% | 主要渠道,分担大部分流量 |
| ✅ 良好 | 8-10 | 12-18% | 备选渠道,分担部分流量 |
| 🔧 一般 | 3-7 | 5-12% | 普通渠道,承担少量流量 |
| 🚨 应急 | 1-2 | 1-5% | 备用渠道,仅作应急 |

### 2.2 权重配置实战

#### 场景1: 平均分配 (最简单)

**适用**: 所有渠道质量差不多

```
配置:
渠道A: 权重 10
渠道B: 权重 10
渠道C: 权重 10

效果:
每个渠道分配 33.3% 流量

优点: 配置简单,负载均衡
缺点: 无法突出优质渠道
```

#### 场景2: 主次分配 (推荐)

**适用**: 有明显的主力和备用渠道

```
配置:
渠道A(优质主力): 权重 20  → 54.5%流量
渠道B(良好备选): 权重 12  → 32.7%流量
渠道C(普通备用): 权重 5   → 12.8%流量

效果:
主力渠道承担过半流量
备选分担大部分剩余流量
备用作为补充

优点: 优质渠道得到充分利用
建议: 最常用的配置方式
```

#### 场景3: 金字塔分配 (高级)

**适用**: 多层次渠道体系

```
配置:
顶级渠道 (2个):
    渠道A: 权重 20  → 25%流量
    渠道B: 权重 20  → 25%流量

主力渠道 (3个):
    渠道C: 权重 12  → 15%流量
    渠道D: 权重 12  → 15%流量
    渠道E: 权重 10  → 12.5%流量

备用渠道 (2个):
    渠道F: 权重 3   → 3.75%流量
    渠道G: 权重 3   → 3.75%流量

总权重: 80
效果:
顶级: 50%流量 (高质量保障)
主力: 42.5%流量 (分担主要流量)
备用: 7.5%流量 (应急备份)

优点: 多层保障,灵活稳定
适用: 大规模运营
```

### 2.3 动态调整策略

**为什么需要动态调整?**

```
问题:
渠道A昨天很好 → 设置高权重
今天突然变慢/不稳定 → 但权重还是高
结果: 大量请求失败,用户体验差

解决:
定期根据数据调整权重
```

**调整依据**:

```
查看数据 (日志→按渠道统计):

渠道A:
  成功率: 99.5%
  平均响应时间: 800ms
  消耗金额: $150
  → 判断: 优秀,提高权重

渠道B:
  成功率: 92%
  平均响应时间: 3500ms
  消耗金额: $80
  → 判断: 一般,降低权重

渠道C:
  成功率: 85%
  平均响应时间: 8000ms
  频繁超时
  → 判断: 差,大幅降低或禁用
```

**调整频率建议**:

```
新渠道: 每天观察和调整 (前1周)
稳定渠道: 每周调整一次
成熟系统: 每2周或每月调整

紧急情况: 立即调整
    - 成功率突降
    - 大量超时
    - 上游通知维护
```

**调整幅度建议**:

```
小幅调整: ±2-5权重
    适用: 微调优化

中幅调整: ±5-10权重
    适用: 性能明显变化

大幅调整: ±10以上
    适用: 质量大幅波动

示例:
渠道A: 权重 15 → 性能提升 → 权重 20
渠道B: 权重 10 → 变慢 → 权重 5
渠道C: 权重 8 → 频繁失败 → 禁用
```

## 🛡️ 三、容灾配置详解

### 3.1 失败重试机制

**重试机制**: 当一个渠道失败时,自动尝试其他渠道。

**配置位置**:

```
设置 → 运营设置 → 通用设置 → 失败重试次数
```

**推荐配置**:

| 渠道数量 | 重试次数 | 说明 |
|---------|---------|------|
| 1-2个 | 0-1次 | 没有足够备份,少重试 |
| 3-5个 | 2-3次 | 有一定备份,适度重试 |
| 6-10个 | 3-5次 | 充足备份,多次重试 |
| 10+个 | 5-7次 | 大量备份,积极重试 |

**注意事项**:

```
✅ 合理设置:
- 重试次数 ≤ (渠道总数 - 1)
- 给予足够尝试机会
- 但避免过度重试(延长响应时间)

❌ 不推荐:
- 只有2个渠道,设置重试10次
- 重试超过可用渠道数
- 重试=0 (关闭重试,单点故障风险)
```

**工作流程**:

```
示例: 3个渠道,重试次数=2

第1次尝试:
请求 → 渠道A → 失败 ❌

第2次尝试 (重试1):
请求 → 渠道B → 失败 ❌

第3次尝试 (重试2):
请求 → 渠道C → 成功 ✅
最终返回结果

如果3次都失败:
返回错误给用户
```

### 3.2 自动禁用机制

**自动禁用**: 当渠道连续失败多次,自动临时禁用该渠道。

**配置**:

```
渠道 → 编辑 → 自动封禁

选项:
- 启用: 开启自动禁用
- 禁用: 关闭自动禁用

默认: 启用
```

**触发条件**:

```
判断逻辑 (代码内置):
IF 连续失败次数 >= 10次 THEN
    自动禁用该渠道
    记录禁用原因和时间
```

**恢复机制**:

```
方式1: 手动恢复
    渠道 → 找到被禁用渠道 → 点击"测试"
    如果测试通过 → 点击"启用"

方式2: 自动恢复 (如果配置了)
    系统定期重新测试被禁用渠道
    如果恢复正常,自动启用

推荐: 手动恢复 (更可控)
```

**最佳实践**:

```
✅ 建议开启自动禁用:
    - 保护系统避免持续失败
    - 减少无效重试
    - 自动隔离问题渠道

⚙️ 配合定期检查:
    - 每天查看被禁用渠道
    - 联系上游排查问题
    - 问题解决后手动启用
```

### 3.3 优先级分层

**优先级**: 控制渠道选择的优先顺序,高优先级的渠道优先被选中。

**配置方法**:

```
渠道 → 编辑 → 优先级

数值: 0 - 999999 (数字越大越优先)

推荐分层:
紧急/VIP渠道: 100
主力渠道: 50
普通渠道: 10
备用渠道: 0
```

**工作机制**:

```
选择逻辑:
1. 按优先级从高到低排序
2. 在同一优先级层内,按权重随机选择
3. 如果当前层所有渠道都失败,尝试下一层

示例:
渠道A: 优先级 100, 权重 10
渠道B: 优先级 100, 权重 20
渠道C: 优先级 50, 权重 15
渠道D: 优先级 0, 权重 5

选择过程:
第1步: 尝试优先级100的渠道
    在A和B之间按权重选择
    B被选中概率 = 20/(10+20) = 67%
    A被选中概率 = 33%

如果B失败:
第2步: 重试优先级100的另一个渠道A

如果A也失败:
第3步: 尝试优先级50的渠道C

如果C也失败:
第4步: 尝试优先级0的渠道D

如果D也失败:
返回错误
```

### 3.4 分组隔离

**分组隔离**: 将渠道分为不同分组,不同分组的渠道互不影响。

**使用场景**:

```
场景1: VIP和普通用户隔离
    VIP分组: 顶级渠道 (优质快速)
    default分组: 普通渠道 (性价比)

    效果:
    - VIP用户不受普通用户影响
    - 保证VIP服务质量

场景2: 按服务等级分组
    premium分组: 高质量服务
    standard分组: 标准服务
    budget分组: 经济服务

场景3: 按用途分组
    production分组: 生产环境
    testing分组: 测试环境
    internal分组: 内部使用
```

**配置方法**:

```
步骤1: 创建分组渠道
    渠道 → 创建 → 分组: vip
    添加优质渠道到vip分组

步骤2: 令牌指定分组
    令牌 → 编辑 → 渠道分组: vip

步骤3: 测试验证
    使用该令牌调用API
    查看日志,确认使用了vip分组的渠道
```

**隔离效果**:

```
配置前:
所有用户共享所有渠道
    → VIP用户可能使用到慢速渠道
    → 普通用户占用优质渠道资源

配置后:
VIP用户 → vip分组渠道 (独享)
普通用户 → default分组渠道 (独享)
    → 互不影响
    → 资源隔离
```

## 🎯 四、高可用方案设计

### 4.1 双备份方案 (入门级)

**适用**: 小规模运营,预算有限

**配置**:

```
渠道设计:
┌──────────────────────────────────┐
│ 主力渠道A: 权重 15, 优先级 50   │
│ 备用渠道B: 权重 10, 优先级 10   │
└──────────────────────────────────┘

系统设置:
- 失败重试次数: 1
- 自动禁用: 开启

工作流程:
请求 → 优先选择A
      → 如果A失败,重试B
      → 如果B也失败,返回错误

可用性: 95-98%
成本: 低
```

**优缺点**:

```
优点:
✅ 配置简单
✅ 成本低
✅ 满足基本容灾需求

缺点:
⚠️ 备份有限,存在单点风险
⚠️ 可用性不够高
⚠️ 不适合高并发场景
```

### 4.2 三级容灾方案 (推荐)

**适用**: 中等规模,追求稳定性

**配置**:

```
渠道设计:
┌────────────────────────────────────────┐
│ 第1层(主力): 2个渠道                   │
│   渠道A: 权重 20, 优先级 100           │
│   渠道B: 权重 15, 优先级 100           │
├────────────────────────────────────────┤
│ 第2层(备选): 2个渠道                   │
│   渠道C: 权重 12, 优先级 50            │
│   渠道D: 权重 10, 优先级 50            │
├────────────────────────────────────────┤
│ 第3层(备用): 1-2个渠道                 │
│   渠道E: 权重 5, 优先级 0              │
└────────────────────────────────────────┘

系统设置:
- 失败重试次数: 3
- 自动禁用: 开启

工作流程:
第1步: 在A和B之间选择 (优先级100)
       A概率 57%, B概率 43%

第2步: 如失败,尝试另一个优先级100渠道

第3步: 如都失败,降级到优先级50
       在C和D之间选择

第4步: 如还失败,尝试优先级0的备用E

可用性: 99%+
成本: 中等
```

**优缺点**:

```
优点:
✅ 三层保障,可用性高
✅ 灵活分配流量
✅ 平衡了成本和稳定性
✅ 适合大多数场景

缺点:
⚠️ 配置稍复杂
⚠️ 需要一定的渠道数量
⚠️ 维护成本相对较高
```

### 4.3 企业级高可用方案

**适用**: 大规模运营,严格SLA要求

**配置**:

```
渠道设计:
┌──────────────────────────────────────────────┐
│ 超级层(VIP专用): 3个顶级渠道                 │
│   权重 20, 优先级 200                         │
├──────────────────────────────────────────────┤
│ 主力层(核心流量): 4个优质渠道                │
│   权重 15-18, 优先级 100                      │
├──────────────────────────────────────────────┤
│ 备选层(分流): 3个良好渠道                    │
│   权重 10-12, 优先级 50                       │
├──────────────────────────────────────────────┤
│ 备用层(应急): 2-3个普通渠道                  │
│   权重 5-8, 优先级 10                         │
├──────────────────────────────────────────────┤
│ 兜底层(最后保障): 1-2个廉价渠道              │
│   权重 3, 优先级 0                            │
└──────────────────────────────────────────────┘

分组策略:
- VIP分组: 超级层 + 主力层
- 高级分组: 主力层 + 备选层
- 普通分组: 备选层 + 备用层
- 测试分组: 备用层 + 兜底层

系统设置:
- 失败重试次数: 5-7
- 自动禁用: 开启
- 重试延迟: 递增 (第1次0ms, 第2次100ms, 第3次200ms...)

监控和告警:
- 实时监控所有渠道状态
- 成功率<95%告警
- 响应时间>5s告警
- 自动禁用渠道通知

可用性目标: 99.9%+
成本: 高
```

**工作流程**:

```
VIP用户请求:
第1步: 优先级200 (超级层)
       3个顶级渠道,按权重选择

第2步: 如失败,重试优先级200的其他渠道

第3步: 如都失败,降级到优先级100 (主力层)
       4个优质渠道,继续尝试

第4步: 如还失败,继续降级...
       最多尝试7次

效果:
- 多达15+个渠道可用
- 故障自动切换
- VIP用户优先保障
- 极高可用性
```

## 📊 五、性能优化和监控

### 5.1 性能数据分析

**关键指标**:

| 指标 | 说明 | 目标值 | 监控方式 |
|------|------|-------|---------|
| **成功率** | 请求成功比例 | >98% | 日志→按渠道统计 |
| **响应时间** | 平均响应延迟 | <2000ms | 渠道列表查看 |
| **错误率** | 失败请求比例 | <2% | 日志→筛选失败 |
| **重试率** | 需要重试的比例 | <5% | 系统日志分析 |
| **渠道利用率** | 各渠道流量分布 | 均衡 | 日志→渠道统计 |

**数据采集方法**:

```
步骤1: 导出日志数据
    日志 → 选择时间范围 → 导出CSV

步骤2: 使用Excel分析
    透视表 → 行:渠道名 → 值:计数,平均响应时间

步骤3: 计算指标
    成功率 = 成功数 / 总请求数
    平均响应时间 = SUM(响应时间) / 请求数
    渠道流量占比 = 渠道请求数 / 总请求数
```

### 5.2 优化决策矩阵

**根据数据做决策**:

| 成功率 | 响应时间 | 决策 |
|-------|---------|------|
| >99% | <1500ms | 🌟 提高权重+10,增加流量 |
| 95-99% | 1500-3000ms | ✅ 保持当前配置 |
| 90-95% | 3000-5000ms | ⚠️ 降低权重-5,减少流量 |
| <90% | >5000ms | ❌ 大幅降低权重或禁用 |

**优化流程**:

```
每周优化检查:

1. 收集数据 (过去7天)
2. 识别优秀渠道:
   - 成功率 top 3
   - 响应时间 top 3
   → 提高权重

3. 识别问题渠道:
   - 成功率 < 95%
   - 响应时间 > 5000ms
   → 降低权重或禁用

4. 更新配置:
   渠道 → 编辑 → 调整权重 → 保存

5. 观察效果 (1-3天)
6. 如有改善,保持配置
   如无改善,继续调整
```

### 5.3 容灾演练

**为什么需要演练?**

```
问题: 配置了容灾,但不确定是否真的有效
风险: 真正故障时才发现配置问题
解决: 定期模拟故障,验证容灾能力
```

**演练步骤**:

```
演练1: 单渠道故障
    1. 记录当前系统状态
    2. 手动禁用1个主力渠道
    3. 发送测试请求100次
    4. 检查:
       - 请求是否全部成功?
       - 自动切换到其他渠道?
       - 用户是否感知到故障?
    5. 重新启用渠道
    6. 验证恢复正常

演练2: 多渠道故障
    1. 同时禁用2个渠道
    2. 发送测试请求
    3. 验证备用渠道接管
    4. 检查重试机制是否正常

演练3: 分组故障
    1. 禁用某个分组的所有渠道
    2. 测试该分组的令牌
    3. 验证是否返回正确错误提示

演练频率:
- 新系统: 每周一次
- 稳定系统: 每月一次
- 重大配置变更后: 立即演练
```

### 5.4 监控和告警设置

**推荐监控方案**:

```
方案1: 内置监控 (基础)
    设置 → 运营设置 → 监控
    - 渠道状态监控
    - 余额监控
    - 异常请求监控

方案2: 外部监控 (进阶)
    工具: Uptime Kuma / UptimeRobot
    - 监控New API服务可用性
    - 监控API端点响应时间
    - 故障告警到邮件/微信

方案3: 日志监控 (专业)
    工具: Grafana + Prometheus
    - 实时监控各项指标
    - 可视化仪表板
    - 自定义告警规则
```

**告警配置建议**:

| 告警级别 | 触发条件 | 告警方式 | 响应时间 |
|---------|---------|---------|---------|
| 🔴 紧急 | 所有渠道失败 | 短信+电话 | 5分钟内 |
| 🟠 重要 | 成功率<90% | 邮件+微信 | 30分钟内 |
| 🟡 提醒 | 响应时间>5s | 邮件 | 2小时内 |
| 🔵 信息 | 渠道被禁用 | 系统通知 | 下一工作日 |

## 🛠️ 六、常见配置案例

### 案例1: 成本优先配置

**需求**: 尽量降低成本,可接受偶尔不稳定

```
渠道配置:
廉价渠道A: 权重 15 (承担主要流量)
廉价渠道B: 权重 12
正常渠道C: 权重 5 (备用,较贵)

容灾配置:
- 重试次数: 2
- 自动禁用: 开启

策略:
- 优先使用便宜渠道
- 失败后才用贵的渠道
- 可用性: 95-97%

适用: 测试环境、内部使用
```

### 案例2: 稳定性优先配置

**需求**: 追求最高稳定性,成本次要

```
渠道配置:
顶级渠道A: 权重 20, 优先级 100
顶级渠道B: 权重 18, 优先级 100
优质渠道C: 权重 15, 优先级 50
优质渠道D: 权重 12, 优先级 50
备用渠道E: 权重 10, 优先级 10

容灾配置:
- 重试次数: 4
- 自动禁用: 开启
- 所有渠道保持充足余额

策略:
- 多个顶级渠道互为备份
- 分层容灾
- 可用性: 99.5%+

适用: 生产环境、付费客户
```

### 案例3: 分时段配置 (高级)

**需求**: 根据流量高峰调整策略

**高峰期配置** (9:00-22:00):

```
快速渠道A: 权重 20 (多花钱买速度)
快速渠道B: 权重 18
普通渠道C: 权重 8

策略: 优先快速渠道,保证用户体验
```

**低谷期配置** (22:00-9:00):

```
便宜渠道D: 权重 18 (省钱)
便宜渠道E: 权重 15
快速渠道A: 权重 8 (备用)

策略: 使用便宜渠道,降低成本
```

**实现方法**:

```
方式1: 手动切换 (简单)
    早上9点: 登录后台,调整权重
    晚上10点: 再次调整

方式2: 定时脚本 (自动化)
    编写脚本定时调用API修改权重
    需要一定开发能力

方式3: 创建两套分组 (推荐)
    peak分组: 快速渠道
    valley分组: 便宜渠道
    不同时段的令牌使用不同分组
```

### 案例4: 地域优化配置

**需求**: 用户分布在不同地区

```
分组设计:
国内分组 (cn):
    国内渠道A: 权重 20
    国内渠道B: 权重 15
    国际渠道C: 权重 5 (备用)

国际分组 (intl):
    国际渠道C: 权重 20
    国际渠道D: 权重 15
    国内渠道A: 权重 5 (备用)

分配策略:
国内用户 → cn分组令牌 → 使用国内渠道(快)
国外用户 → intl分组令牌 → 使用国际渠道(快)

效果:
- 降低网络延迟
- 提高响应速度
- 改善用户体验
```

## 📈 七、实战优化流程

### 7.1 初始配置 (第1周)

```
Day 1: 基础配置
    - 添加2-3个渠道
    - 所有权重设为10 (平均分配)
    - 重试次数设为2
    - 开启自动禁用

Day 2-7: 收集数据
    - 每天查看日志
    - 记录各渠道表现
    - 观察哪些渠道好,哪些不好
```

### 7.2 第一次优化 (第2周)

```
基于第1周数据:

步骤1: 数据分析
    导出7天日志数据
    统计各渠道:
    - 请求数
    - 成功率
    - 平均响应时间

步骤2: 渠道评级
    S级 (优秀): 成功率>99%, 响应<1500ms
    A级 (良好): 成功率95-99%, 响应1500-3000ms
    B级 (一般): 成功率90-95%, 响应3000-5000ms
    C级 (差): 成功率<90%, 响应>5000ms

步骤3: 调整权重
    S级渠道: 权重提高到18-20
    A级渠道: 权重调整为12-15
    B级渠道: 权重降低到5-8
    C级渠道: 降低到3或禁用

步骤4: 观察效果 (1周)
```

### 7.3 持续优化 (第3周及之后)

```
每周检查:
1. 查看过去7天数据
2. 对比上周,识别变化
3. 调整权重 (微调±2-5)
4. 记录调整原因和效果

每月大优化:
1. 全面分析1个月数据
2. 淘汰持续表现差的渠道
3. 测试并添加新渠道
4. 重新规划权重分配
5. 更新容灾策略

效果:
3个月后,系统达到最优状态:
- 可用性>99%
- 成本降低20-30%
- 响应时间缩短30-50%
```

## 🎯 八、故障应对指南

### 8.1 常见故障场景

#### 场景1: 单个渠道突然大量失败

**症状**:

```
日志中某个渠道红色错误爆增
其他渠道正常
```

**应对步骤**:

```
第1步: 立即禁用该渠道
    渠道 → 找到故障渠道 → 状态改为"禁用"

第2步: 检查系统状态
    仪表板 → 查看成功率是否恢复正常
    日志 → 确认新请求使用其他渠道

第3步: 排查原因
    测试该渠道 → 查看错误信息
    登录上游平台 → 检查:
    - 余额是否不足
    - 是否在维护
    - 密钥是否失效
    - 是否被限流或封禁

第4步: 解决问题
    - 如余额不足 → 充值
    - 如维护中 → 等待维护结束
    - 如密钥失效 → 更新密钥
    - 如被封禁 → 联系上游客服

第5步: 恢复服务
    问题解决后:
    渠道 → 测试 → 确认通过 → 启用
```

#### 场景2: 所有渠道集体故障

**症状**:

```
仪表板成功率骤降
用户大量反馈无法使用
日志全是失败记录
```

**应对步骤**:

```
第1步: 确认问题范围
    是所有用户都失败?还是部分?
    是所有模型都失败?还是特定模型?

第2步: 检查系统本身
    - New API服务是否正常运行?
    - 服务器资源是否充足?
    - 数据库是否正常?
    - 网络是否畅通?

第3步: 检查上游
    逐个测试上游平台:
    - 能否正常访问?
    - 控制台是否显示正常?
    - 是否有维护公告?

第4步: 紧急应对
    如果是系统问题:
    - 重启New API服务
    - 检查日志排查具体原因

    如果是上游集体故障:
    - 发布公告告知用户
    - 寻找临时替代方案
    - 联系上游了解恢复时间

第5步: 事后总结
    - 分析故障原因
    - 改进监控和告警
    - 增加冗余备份
```

#### 场景3: 部分用户无法使用

**症状**:

```
部分用户报错
其他用户正常
渠道状态都正常
```

**排查步骤**:

```
第1步: 检查用户账户
    用户 → 搜索该用户
    - 额度是否充足?
    - 状态是否正常?

第2步: 检查令牌
    令牌 → 搜索该用户令牌
    - 是否过期?
    - 额度是否用完?
    - 是否被禁用?
    - IP限制是否正确?

第3步: 检查日志
    日志 → 筛选该用户
    查看具体错误信息:
    - 401: 令牌无效
    - 403: 无权限
    - 429: 超出频率限制
    - 余额不足: 充值问题

第4步: 针对性解决
    根据具体原因处理
```

### 8.2 故障预防措施

```
预防措施清单:

✅ 渠道层面:
- 至少配置3个以上渠道
- 不同供应商分散风险
- 所有渠道保持充足余额
- 定期测试所有渠道

✅ 配置层面:
- 开启自动禁用机制
- 设置合理的重试次数
- 配置多层优先级
- 使用分组隔离风险

✅ 监控层面:
- 设置实时监控告警
- 每日检查系统状态
- 定期分析日志数据
- 建立应急响应流程

✅ 备份层面:
- 定期备份数据库
- 保留配置备份
- 准备应急方案
- 建立供应商联系清单
```
