# New API 安全和防封号策略指南

## 📋 文档概述

本文档详细说明如何保护你的上游API账号不被封禁,以及如何防止系统被滥用。

### 为什么重要?

```
问题场景:
你买了一个上游API账号 → 成本$100
配置到New API使用 → 运行正常
几天后账号被封 → $100打水漂
需要重新购买 → 持续损失

如果频繁被封:
- 经济损失巨大
- 服务频繁中断
- 用户体验极差
- 运营成本高企
```

### 本文将解决

- ✅ 为什么会被封号?
- ✅ 如何配置代理防封?
- ✅ 如何控制请求频率?
- ✅ 如何启用粘性会话?
- ✅ 如何防止令牌被盗用?
- ✅ 完整的防封号策略

## 🔍 一、封号原因分析

### 1.1 常见封号原因

| 原因 | 风险等级 | 检测方式 | 说明 |
|------|---------|---------|------|
| **多账号同IP** | 🔴 极高 | IP关联检测 | 多个API Key从同一IP请求 |
| **高频请求** | 🔴 高 | 频率限制 | 短时间大量请求 |
| **异常行为模式** | 🟠 中高 | 行为分析 | 24小时不间断请求 |
| **数据中心IP** | 🟠 中 | IP类型检测 | 使用服务器IP而非住宅IP |
| **会话不一致** | 🟡 中低 | 会话跟踪 | 同一对话频繁切换IP |
| **地区异常** | 🟡 低 | 地理位置 | IP地理位置频繁变化 |

### 1.2 封号检测机制

**上游平台的风控手段**:

```
第1层: IP检测
    检查IP类型:
    - 住宅IP (安全) ✅
    - 数据中心IP (可疑) ⚠️
    - 已知VPN/代理IP (高危) ❌

    检查IP行为:
    - 单IP请求量是否异常
    - 是否有多个账号从同一IP
    - IP地理位置是否合理

第2层: 频率检测
    统计指标:
    - RPM (每分钟请求数)
    - QPD (每日请求量)
    - 并发连接数

    异常判断:
    - 超过官方限制
    - 短时间暴增
    - 明显的机器人行为

第3层: 行为分析
    模式识别:
    - 请求时间分布 (24小时均匀 = 可疑)
    - 会话连续性 (频繁跳跃 = 异常)
    - 请求内容相似度 (重复测试 = 机器人)

第4层: 账号关联
    多维度关联:
    - 注册邮箱相似
    - 支付方式相同
    - 登录IP重叠
    - 使用模式一致
```

### 1.3 风险等级评估

**自查表**: 评估你的当前配置风险

```
检查项目:

□ 使用了代理? (没有 +50分风险)
□ 代理类型是? (数据中心IP +30分, 住宅IP +5分)
□ 多个渠道共用代理? (是 +40分)
□ 开启粘性会话? (没有 +20分)
□ 设置了请求频率限制? (没有 +30分)
□ 有异常高频用户? (有 +25分)
□ 渠道数量? (1个 +30分, 2-3个 +15分)

总分评估:
0-30分: 🟢 风险较低
31-60分: 🟡 中等风险,需要改进
61-100分: 🟠 高风险,建议立即优化
100+分: 🔴 极高风险,随时可能被封
```

## 🛡️ 二、代理配置防封 (最重要!)

### 2.1 为什么必须使用代理?

**不使用代理的后果**:

```
场景: 你有3个上游API账号

没有代理:
所有渠道 → 服务器IP: 123.45.67.89
上游检测: 发现3个账号都来自同一IP
结果: 识别为批量账号 → 全部封禁 ❌

使用代理:
渠道A → 代理IP1: 10.1.1.1
渠道B → 代理IP2: 10.2.2.2
渠道C → 代理IP3: 10.3.3.3
上游检测: 看到3个不同的用户IP
结果: 正常用户 → 安全 ✅
```

### 2.2 代理类型选择

| 代理类型 | 安全性 | 速度 | 成本 | 推荐度 | 说明 |
|---------|-------|------|------|--------|------|
| **静态住宅IP** | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ | 💰💰💰 | 🌟最佳 | 固定的住宅网络IP,最接近真实用户 |
| **动态住宅IP** | ⭐⭐⭐⭐ | ⭐⭐⭐ | 💰💰 | ✅推荐 | IP会定期更换,安全性好 |
| **ISP代理** | ⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | 💰💰💰 | ✅推荐 | 企业宽带IP,速度快且稳定 |
| **数据中心IP** | ⭐⭐ | ⭐⭐⭐⭐⭐ | 💰 | ⚠️谨慎 | 服务器IP,容易被识别 |
| **免费VPN/代理** | ⭐ | ⭐⭐ | 免费 | ❌不推荐 | IP被滥用,高风险 |

**推荐供应商** (参考):

```
注意: 以下仅供参考,不构成推荐

住宅代理:
- Bright Data (贵但质量好)
- Oxylabs
- Smartproxy
- IPRoyal

国内选择:
- 自建Clash代理池
- 购买稳定的机场节点
- 某些中转API本身提供代理
```

### 2.3 代理配置方法

#### 方法1: 渠道级代理配置 (推荐)

**为每个渠道配置独立代理**:

```
步骤1: 准备代理
    购买或准备代理服务器
    获取代理地址,例如:
    - HTTP: http://proxy1.example.com:8080
    - SOCKS5: socks5://proxy2.example.com:1080

步骤2: 配置渠道
    渠道 → 编辑 → 其他设置

    填入JSON配置:
    {
      "proxy": "http://proxy1.example.com:8080"
    }

    如需认证:
    {
      "proxy": "http://username:password@proxy1.example.com:8080"
    }

步骤3: 为不同渠道配置不同代理
    渠道A → proxy1
    渠道B → proxy2
    渠道C → proxy3

步骤4: 测试
    点击"测试"按钮
    确认通过代理连接成功
```

**完整配置示例**:

```json
// 渠道A - 使用HTTP代理
{
  "proxy": "http://user:pass@proxy-us-1.example.com:8080"
}

// 渠道B - 使用SOCKS5代理
{
  "proxy": "socks5://proxy-eu-1.example.com:1080"
}

// 渠道C - 使用认证的HTTPS代理
{
  "proxy": "https://user:pass@proxy-asia-1.example.com:443"
}
```

#### 方法2: 使用Clash代理池

**适合**: 有技术能力,预算有限

**原理**:

```
购买机场订阅 → Clash客户端 → 代理池
New API → Clash本地端口 → 自动选择节点

优点:
- 成本较低 (机场订阅便宜)
- IP自动轮换
- 节点丰富

缺点:
- 需要维护Clash
- 稳定性不如专业代理
- 需要一定技术能力
```

**配置步骤**:

```
步骤1: 安装Clash
    服务器上安装Clash客户端
    配置订阅链接

步骤2: 启用本地代理端口
    Clash配置中设置:
    mixed-port: 7890 (HTTP+SOCKS5混合端口)

步骤3: 配置New API渠道
    {
      "proxy": "http://127.0.0.1:7890"
    }

步骤4: 配置Clash规则
    选择策略组:
    - 负载均衡 (推荐)
    - 自动选择
    - 手动选择

效果:
所有渠道通过Clash → 自动分配不同节点
```

#### 方法3: 系统级代理 (不推荐)

**为什么不推荐?**

```
问题:
所有渠道共用同一个系统代理
    ↓
所有API Key从同一IP发出请求
    ↓
被识别为关联账号
    ↓
一起被封 ❌

正确做法:
每个渠道独立配置代理 ✅
```

### 2.4 代理配置最佳实践

**原则1: 一渠道一IP**

```
✅ 正确:
渠道A → 代理IP 1.1.1.1
渠道B → 代理IP 2.2.2.2
渠道C → 代理IP 3.3.3.3

❌ 错误:
渠道A、B、C → 同一代理IP 1.1.1.1
```

**原则2: 使用住宅IP**

```
✅ 优先级:
1. 静态住宅IP (最佳)
2. 动态住宅IP
3. ISP代理
4. 优质数据中心IP

❌ 避免:
- 免费代理
- 公共VPN
- 已知被滥用的IP
```

**原则3: 长期稳定使用**

```
✅ 推荐:
同一个渠道长期使用同一个代理IP
    → 模拟真实用户行为
    → 降低风控触发

❌ 避免:
频繁更换代理IP
    → 行为异常
    → 容易触发风控
```

**原则4: 地理位置合理**

```
如果你的业务主要在:
- 中国 → 使用中国大陆或香港IP
- 美国 → 使用美国IP
- 欧洲 → 使用欧洲IP

避免:
业务在中国,使用非洲IP (不合理)
```

## ⚡ 三、请求频率控制

### 3.1 为什么要限制频率?

**过高频率的危害**:

```
正常用户:
每分钟 10-30 次请求

机器人/滥用:
每分钟 500+ 次请求

上游检测:
发现异常高频 → 判定为滥用 → 封号
```

### 3.2 频率限制配置

#### 令牌级别限制 (推荐)

**配置位置**: 令牌 → 编辑 → 高级设置

| 限制类型 | 说明 | 推荐值 | 配置字段 |
|---------|------|-------|---------|
| **RPM** | 每分钟请求数 | 60-120 | 请求频率限制 |
| **TPM** | 每分钟Token数 | 50000-100000 | Token频率限制 |
| **RPD** | 每日请求数 | 5000-10000 | 每日请求限制 |
| **并发数** | 同时进行的请求 | 3-10 | 并发限制 |

**配置示例**:

```
普通用户令牌:
- RPM: 60 (每分钟60次)
- TPM: 50000 (每分钟5万Token)
- 并发: 5

VIP用户令牌:
- RPM: 120 (更宽松)
- TPM: 100000
- 并发: 10

测试令牌:
- RPM: 10 (限制更严)
- TPM: 10000
- 并发: 2
```

#### 用户级别限制

**配置位置**: 用户 → 编辑 → 额度和限制

```
配置项:
- 最大并发请求数
- 每日最大请求数
- 每日最大消费额度

示例:
用户A (重度用户):
  并发: 10
  每日请求: 10000
  每日消费: $50

用户B (轻度用户):
  并发: 3
  每日请求: 1000
  每日消费: $10
```

### 3.3 动态限流策略

**根据时段调整**:

```
高峰期 (9:00-22:00):
    更严格限制,避免过载
    RPM: 60
    并发: 5

低谷期 (22:00-9:00):
    放宽限制
    RPM: 100
    并发: 10

实现:
方式1: 手动调整 (简单但需人工)
方式2: 脚本自动调整 (需开发能力)
```

**根据用户信用调整**:

```
新用户:
    严格限制,观察期
    RPM: 30
    并发: 3

正常用户 (使用1月+,无异常):
    正常限制
    RPM: 60
    并发: 5

VIP用户 (大额充值,信任度高):
    宽松限制
    RPM: 120
    并发: 10
```

### 3.4 异常检测和处理

**检测异常高频用户**:

```
监控指标:
- 请求量突增 (>平时3倍)
- 短时间大量请求 (5分钟>300次)
- 频繁触发限流
- 大量相似请求

检测方式:
日志 → 按用户统计 → 排序请求量 → 识别Top异常用户
```

**处理流程**:

```
第1步: 自动限流
    触发限流后:
    - 后续请求返回 429 错误
    - 冷却时间: 5-10分钟

第2步: 告警通知
    异常用户触发告警:
    - 管理员收到通知
    - 记录用户ID和行为

第3步: 人工审查
    审查用户:
    - 是否是正常业务高峰?
    - 是否是恶意攻击?
    - 是否令牌被盗用?

第4步: 处理决策
    正常高峰: 临时提高限制
    疑似盗用: 暂停令牌,联系用户
    恶意攻击: 封禁用户和令牌
```

## 🔄 四、粘性会话配置

### 4.1 什么是粘性会话?

**粘性会话** (Sticky Session): 同一对话的多次请求使用同一个渠道。

**为什么需要?**

```
没有粘性会话:
用户对话:
  请求1 "你好" → 渠道A (IP: 1.1.1.1) → 回复
  请求2 "继续" → 渠道B (IP: 2.2.2.2) → 回复
  请求3 "再说" → 渠道C (IP: 3.3.3.3) → 回复

上游检测:
  同一对话,IP频繁跳跃
    → 异常行为
    → 触发风控 ⚠️

有粘性会话:
用户对话:
  请求1 "你好" → 渠道A (IP: 1.1.1.1) → 回复
  请求2 "继续" → 渠道A (IP: 1.1.1.1) → 回复
  请求3 "再说" → 渠道A (IP: 1.1.1.1) → 回复

上游检测:
  同一对话,同一IP
    → 正常用户行为
    → 安全 ✅
```

### 4.2 粘性会话配置 (计划中功能)

**注意**: New API目前**暂不支持**粘性会话功能,但系统已有相关设计。

**预计配置方式** (参考已有的技术设计):

```
令牌 → 编辑 → 高级设置

配置项:
□ 启用粘性会话
  会话TTL: [3600] 秒 (1小时)

说明:
- 启用后,同一user字段的请求使用同一渠道
- TTL: 会话保持时间,过期后允许切换
- 渠道失败时自动切换到其他渠道
```

**工作原理** (预期):

```
第1次请求:
user: "user_123"
    ↓
系统选择: 渠道A
    ↓
记录绑定: user_123 → 渠道A (过期时间: 当前时间+3600秒)

第2-N次请求:
user: "user_123"
    ↓
查询绑定: 发现 user_123 → 渠道A
    ↓
继续使用: 渠道A

渠道失败:
渠道A请求失败
    ↓
解除绑定: user_123 ⇸ 渠道A
    ↓
重新选择: 渠道B
    ↓
新绑定: user_123 → 渠道B
```

### 4.3 临时替代方案

在官方功能上线前,可以这样降低风险:

**方案1: 降低渠道数量**

```
策略:
使用较少的渠道 (2-3个)
    → 请求自然更多地集中在少数渠道
    → 减少IP跳跃频率

缺点: 降低了负载均衡效果
```

**方案2: 增加渠道权重差异**

```
策略:
主力渠道: 权重 30 (70%流量)
备用渠道: 权重 10 (30%流量)
    → 大部分请求走主力渠道
    → 同一用户更可能持续使用同一渠道

缺点: 主力渠道负载较高
```

**方案3: 使用用户分组**

```
策略:
为不同用户分配不同渠道分组
    用户A → 分组1 (渠道A、B)
    用户B → 分组2 (渠道C、D)

    → 每个用户固定使用特定几个渠道
    → 减少跳跃

配置:
1. 创建多个渠道分组
2. 为不同用户的令牌指定不同分组
```

## 🔐 五、令牌安全保护

### 5.1 令牌配置安全要点

**基础安全配置**:

```
创建令牌时必须配置:

✅ 过期时间
    - 不要设置永久有效
    - 推荐: 30-90天
    - VIP用户可适当延长

✅ 额度限制
    - 单个令牌额度不要过高
    - 推荐: 普通用户 $10-50
    - 即使被盗,损失可控

✅ IP白名单 (如果用户IP固定)
    - 限制只能从特定IP访问
    - 最安全的保护方式
    - 适用于服务器端调用

✅ 请求频率限制
    - RPM: 60
    - TPM: 50000
    - 并发: 5
    - 防止被盗用后疯狂消耗
```

**高级安全配置**:

```
□ 启用用户认证
    要求用户必须登录才能获取令牌

□ 启用2FA (两步验证)
    管理员账户启用2FA
    防止后台被入侵

□ 定期轮换
    定期更换令牌
    推荐: 每3-6个月换一次

□ 访问日志审计
    定期查看访问日志
    识别异常使用模式
```

### 5.2 检测令牌被盗用

**异常特征**:

```
疑似被盗用的信号:

⚠️ 流量突增
    平时每天10次请求
    突然变成每天1000次
    → 可能被盗用

⚠️ 异常时间
    用户平时只在工作时间使用
    突然凌晨3点大量请求
    → 可能被盗用

⚠️ 异常IP
    用户一直从中国访问
    突然出现俄罗斯IP
    → 可能被盗用

⚠️ 异常内容
    平时请求正常业务
    突然大量重复/测试请求
    → 可能被盗用
```

**检测方法**:

```
步骤1: 查看用户统计
    日志 → 按用户分组
    排序: 按请求数降序
    识别: 请求数异常高的用户

步骤2: 查看详细日志
    日志 → 筛选该用户
    查看:
    - 请求时间分布
    - 请求IP地址
    - 请求内容模式

步骤3: 对比历史数据
    对比最近7天 vs 之前30天
    识别: 异常变化

步骤4: 用户询问
    联系用户确认是否是其本人使用
```

**处理流程**:

```
第1步: 立即暂停令牌
    令牌 → 编辑 → 状态: 禁用

第2步: 联系用户
    通知用户令牌可能被盗用
    要求用户确认

第3步: 调查原因
    - 令牌是否泄露在公开代码?
    - GitHub等平台是否误提交?
    - 是否被他人获取?

第4步: 重新分配
    为用户生成新令牌
    建议用户检查代码安全
```

### 5.3 防止泄露最佳实践

**开发者用户教育**:

```
提供给用户的安全指南:

✅ 不要在代码中硬编码令牌
    ❌ const API_KEY = "sk-xxxxx"
    ✅ const API_KEY = process.env.API_KEY

✅ 不要提交到Git
    添加到 .gitignore:
    .env
    config.json

✅ 使用环境变量
    .env 文件:
    API_KEY=sk-xxxxx

    代码中:
    const apiKey = process.env.API_KEY

✅ 不要在前端直接使用
    前端 → 你的后端 → New API
    不要: 前端 → 直接调用New API
```

### 7.1 账号被封后处理

**立即行动**:

```
第1步: 隔离影响

    - 如果服务中断,发布公告
    - 说明情况和恢复时间
    - 保持透明沟通

第3步: 分析原因
    - 查看该渠道最近的使用日志
    - 识别触发封号的可能原因:
      * IP问题?
      * 频率问题?
      * 异常请求?

第4步: 联系上游
    - 如果是误封,提交申诉
    - 提供使用证明
    - 请求恢复账号

第5步: 补充渠道
    - 购买新账号
    - 配置更好的防护
    - 测试后上线
```

## ❓ 常见问题

**Q: 一定要配置代理吗?**

A: **强烈推荐配置代理**,尤其是多个渠道的情况。不配置代理最大的风险是多账号关联被封。

**Q: 什么代理最好?**

A: 优先级: 静态住宅IP > 动态住宅IP > ISP代理 > 数据中心IP。住宅IP最安全,但成本较高。

**Q: RPM设置多少合适?**

A: 推荐**60-120**。太高容易被风控,太低影响用户体验。根据实际情况调整。

**Q: 已经被封号了怎么办?**

A:
1. 立即禁用该渠道
2. 启用备用渠道
3. 分析被封原因
4. 尝试申诉
5. 采购新账号并做好防护

**Q: 免费代理可以用吗?**

A: **不推荐**。免费代理通常IP质量差,已被大量滥用,风险极高。宁可不用代理,也不要用免费代理。

**Q: 粘性会话什么时候上线?**

A: 目前系统暂不支持,但已有技术设计方案。可关注项目更新。

**Q: 小规模运营需要这么复杂吗?**

A: 至少要做到**Level 1基础防护**:
- 独立代理
- 频率限制
- 多渠道备份
这是基本的安全底线。
