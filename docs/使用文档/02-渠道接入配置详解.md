# New API 渠道接入配置详解

## 📋 文档概述

本文档详细说明如何将**其他中转API站点**的服务接入到 New API 平台,面向非开发人员,提供详细的配置步骤和案例。

### 适用场景
- ✅ 你的上游是**国内中转API站点** (如API2D、AIProxy等)
- ✅ 你的上游不是直接的模型厂商 (OpenAI、Anthropic等)
- ✅ 上游提供的是**OpenAI兼容格式**的API接口

---

## 🎯 一、渠道接入前的准备

### 1.1 理解渠道的概念

**什么是渠道?**

简单来说,一个渠道 = 一个上游供应商的API账号连接。

```
示例:
你在A站买了账号 → 在New API中添加为"渠道A"
你在B站买了账号 → 在New API中添加为"渠道B"
你的用户调用 → New API自动选择渠道A或B → 转发到上游
```

**为什么需要多个渠道?**
- 🔄 负载均衡: 流量分散到多个供应商
- 🛡️ 容灾备份: 一个挂了还有其他的
- 💰 成本优化: 选择性价比高的供应商
- 🚀 性能优化: 选择快速稳定的供应商

### 1.2 收集上游信息清单

在添加渠道前,你需要从上游供应商处获取以下信息:

**必需信息**:
- ✅ **API Key** (密钥) - 例如: `sk-xxxxxxxxxxxxx`
- ✅ **API地址** (BaseURL) - 例如: `https://api.example.com`

**可选信息**:
- 🔸 支持的模型列表 (通常在上游文档中)
- 🔸 请求频率限制 (RPM/TPM)
- 🔸 是否需要其他参数 (如Organization ID)

**如何获取这些信息?**

大多数中转站点会在控制台提供这些信息:

```
登录上游平台 → 控制台/API管理 → 复制API Key和API地址

常见位置:
- "API Key" / "密钥" / "令牌"
- "API地址" / "接口地址" / "Base URL"
- "使用文档" / "开发文档"
```

## 🚀 二、添加渠道的标准流程

### 2.1 添加第一个渠道 (入门)

**场景**: 你在某个中转站点(假设叫"XXX AI")购买了服务。

#### 步骤1: 进入渠道管理页面

```
New API后台 → 左侧菜单"渠道" → 点击右上角"创建新渠道"
```

#### 步骤2: 选择渠道类型

**关键选择**: 渠道类型

| 如果上游是... | 选择类型 |
|-------------|---------|
| OpenAI格式API (最常见) | **OpenAI** |
| 明确标注为Azure OpenAI | **Azure** |
| Claude原生格式 | **Anthropic** |
| Gemini格式 | **Gemini** |
| 不确定/其他格式 | **Custom (自定义)** |

**90%的国内中转站点选择"OpenAI"类型!**

#### 步骤3: 填写基本信息

**表单字段说明**:

| 字段名 | 说明 | 示例 | 是否必填 |
|-------|------|------|----------|
| **类型** | 渠道类型 | OpenAI | ✅ 必填 |
| **名称** | 渠道名称,自定义 | XXX-AI-主力 | ✅ 必填 |
| **分组** | 渠道分组 | default | ✅ 必填 |
| **密钥** | 上游提供的API Key | sk-xxxx | ✅ 必填 |
| **代理** | Base URL地址 | https://api.xxx.com | ✅ 必填 |
| **模型** | 支持的模型,逗号分隔 | gpt-4,gpt-3.5-turbo | ✅ 必填 |
| **权重** | 流量分配权重 | 10 | 可选 |
| **优先级** | 选择优先级 | 0 | 可选 |

**详细填写示例**:

```
类型: OpenAI (下拉选择)
名称: XXX-AI-Channel-1 (方便识别)
分组: default (默认即可,后面会讲分组)
密钥: sk-proj-abc123def456... (从上游复制粘贴)
代理: https://api.xxx-ai.com (上游提供的API地址)
模型: gpt-4,gpt-4-turbo,gpt-3.5-turbo,claude-3-opus (填上游支持的模型,用英文逗号分隔)
权重: 10 (先用默认值)
```

#### 步骤4: 配置额外设置 (重要!)

点击"额外设置"展开,这里有几个关键配置:

**代理配置** (防封必备):

```json
{
  "proxy": "socks5://your-proxy-server:1080"
}
```

如果你有代理服务器(强烈推荐!),在"其他设置"中填写:

```json
{
  "proxy": "http://proxy.example.com:8080"
}
```

或使用socks5代理:

```json
{
  "proxy": "socks5://127.0.0.1:1080"
}
```

**注意**:
- `http://` 或 `socks5://` 前缀必须保留
- 地址要填写完整,包括端口号

#### 步骤5: 测试渠道

**配置完成后,先测试再保存!**

1. 在"测试模型"下拉框选择一个模型 (例如 gpt-3.5-turbo)
2. 点击"测试"按钮
3. 等待3-10秒

**测试结果判断**:

```
✅ 成功: 显示绿色勾号,响应时间,说明配置正确
❌ 失败: 显示红色叉号和错误信息

常见失败原因:
- "Unauthorized" → API Key填错了
- "Not Found" → API地址填错了
- "timeout" → 网络问题或地址错误
- "模型不存在" → 选择的测试模型上游不支持
```

#### 步骤6: 保存并启用

```
1. 确认测试通过
2. 点击"提交"按钮
3. 返回渠道列表,看到新渠道
4. 状态栏显示绿色勾号 = 可用
```

### 2.2 添加更多渠道 (扩展)

重复上述流程,添加更多上游供应商。

**建议配置**:
- 至少2个渠道 (互为备份)
- 最好3-5个渠道 (更好的负载均衡)
- 不同供应商混合 (避免单点故障)

**命名规范建议**:

```
格式: [供应商名]-[特征]-[序号]

示例:
- API2D-主力-1
- AIProxy-备用-2
- XXX-快速-3
- YYY-便宜-4
```

#### 常见问题

**Q: 为什么测试总是失败?**

A: 检查清单:
1. API Key格式正确? (sk-ant-api03-开头)
2. 代理配置了吗? (国内必须配置)
3. 代理能访问Claude吗? (测试: `curl -x proxy https://api.anthropic.com`)
4. 模型名称正确吗? (完整名称,不能简写)
5. 账号余额充足吗?

**Q: 为什么请求很慢?**

A: Claude API响应确实比OpenAI慢:
- Haiku: 1-3秒
- Sonnet: 3-8秒
- Opus: 5-15秒

如果超过这个时间,检查:
- 代理速度
- 网络延迟
- 选择更快的代理节点

**Q: 如何降低被封风险?**

A: 关键措施:
1. **必须使用代理** (住宅IP最佳)
2. 每个Claude渠道使用独立代理IP
3. 启用粘性会话 (功能上线后)
4. 严格限制请求频率 (RPM < 60)
5. 避免多账号关联

详细防封策略请看 `04-安全防封号策略.md`

## 🔧 四、渠道配置高级选项详解

### 4.1 渠道分组 (Group)

**什么是分组?**

分组允许你将渠道分类,给不同用户分配不同分组的渠道。

**使用场景**:

```
场景1: VIP用户专线
    分组: vip
    └─ 添加优质、快速、稳定的渠道
    └─ 仅VIP用户的令牌使用此分组

场景2: 普通用户
    分组: default
    └─ 添加性价比高的渠道
    └─ 普通用户使用

场景3: 测试环境
    分组: test
    └─ 添加测试用渠道
    └─ 内部测试使用
```

**如何使用分组?**

```
步骤1: 创建渠道时选择分组
    渠道 → 创建 → 分组: 输入"vip"或"default"

步骤2: 令牌指定分组
    令牌 → 编辑 → 渠道分组: 选择"vip"

效果:
    该令牌的请求只会路由到"vip"分组的渠道
```

### 4.2 权重 (Weight)

**权重决定流量分配比例**。

**计算公式**:

```
某渠道流量占比 = 该渠道权重 / 所有渠道权重之和

示例:
渠道A: 权重 20
渠道B: 权重 10
渠道C: 权重 5
总权重: 35

流量分配:
渠道A: 20/35 ≈ 57%
渠道B: 10/35 ≈ 29%
渠道C: 5/35 ≈ 14%
```

**权重设置建议**:

| 渠道质量 | 权重值 | 说明 |
|---------|-------|------|
| 🌟 优质主力 | 15-20 | 快速、稳定、余额充足 |
| ⚡ 良好备选 | 8-12 | 性能较好,作为主要分流 |
| 🔧 普通备用 | 3-7 | 性能一般,分担部分流量 |
| 🚨 应急备份 | 1-2 | 不稳定,仅作应急备份 |

**动态调整策略**:

```
初始配置: 所有渠道权重10 (平均分配)
↓
运行1周后,观察数据:
    渠道A: 成功率99%, 响应时间800ms → 权重调整为20
    渠道B: 成功率95%, 响应时间1500ms → 权重调整为10
    渠道C: 成功率88%, 响应时间3000ms → 权重调整为3

效果: 优质渠道承担更多流量
```

### 4.3 优先级 (Priority)

**优先级控制渠道选择顺序**。

**工作原理**:

```
选择渠道时:
1. 先按优先级排序 (数字越大越优先)
2. 在同优先级内按权重随机选择

示例:
渠道A: 优先级 100, 权重 10
渠道B: 优先级 100, 权重 20
渠道C: 优先级 50, 权重 30
渠道D: 优先级 0, 权重 10

选择顺序:
第1层: A和B (优先级100)
    └─ B的概率 = 20/(10+20) = 67%
    └─ A的概率 = 10/(10+20) = 33%
第2层: C (优先级50,仅当第1层都失败时)
第3层: D (优先级0,仅当前面都失败时)
```

**使用场景**:

```
场景1: 官方直连优先
    官方API: 优先级 100
    中转API: 优先级 0
    效果: 优先使用官方,失败后才用中转

场景2: 大客户专线
    VIP渠道: 优先级 100
    普通渠道: 优先级 50
    效果: VIP分组优先使用高质量渠道
```

**建议配置**:

```
简单场景: 所有渠道优先级设为0 (仅用权重)
复杂场景:
    核心渠道: 优先级 100
    备用渠道: 优先级 50
    应急渠道: 优先级 0
```

### 4.4 模型映射 (Model Mapping)

**场景**: 上游的模型名称与你想提供给用户的不同。

**示例**:

```
问题:
上游模型名: gpt-4-turbo-2024-04-09 (太长,用户难记)
你想提供: gpt-4-turbo (简短易记)

解决: 使用模型映射

配置:
渠道 → 编辑 → 模型映射

{
  "gpt-4-turbo": "gpt-4-turbo-2024-04-09",
  "gpt-4": "gpt-4-0613",
  "gpt-3.5": "gpt-3.5-turbo-16k"
}

效果:
用户请求"gpt-4-turbo" → 系统自动转换为上游的"gpt-4-turbo-2024-04-09"
```

**JSON格式**:

```json
{
  "用户使用的模型名": "上游实际的模型名",
  "simple-name": "complex-upstream-name"
}
```

### 4.5 其他设置 (Setting)

这是一个JSON配置,支持多种高级功能。

**完整配置示例**:

```json
{
  "proxy": "http://proxy.example.com:8080",
  "force_format": true,
  "thinking_to_content": false
}
```

**字段说明**:

| 字段 | 类型 | 说明 | 示例 |
|------|------|------|------|
| `proxy` | 字符串 | 代理服务器地址 | `"socks5://127.0.0.1:1080"` |
| `force_format` | 布尔 | 强制转换为OpenAI格式 | `true` / `false` |
| `thinking_to_content` | 布尔 | 将思考内容转为`<think>`标签 | `true` / `false` |

**使用场景**:

**场景1: 配置代理防封号**
```json

  "proxy": "socks5://proxy-server.com:1080"
}
```

**场景2: 非标准格式转换**
```json
{
  "force_format": true

```

适用于上游返回格式不标准,需要强制转换为OpenAI格式。

**场景3: 思考内容转换** (用于o1等模型)
```json

}
```

将模型的思考过程(`reasoning_content`)转换为`<think>`标签包裹在内容中。

**详细说明**:

**权重 (Weight)**:
```
作用: 控制流量分配比例
取值: 0-999999 (0表示禁用,不参与负载均衡)

计算规则:
渠道流量占比 = 该渠道权重 / 所有渠道总权重

示例:
渠道A: 权重 20 → 占比 20/(20+10+5) = 57%
渠道B: 权重 10 → 占比 10/(20+10+5) = 29%
渠道C: 权重 5  → 占比 5/(20+10+5)  = 14%

推荐配置:
优质渠道: 15-20
良好渠道: 10-12
普通渠道: 5-8
备用渠道: 1-3
禁用: 0

详见: 03-容灾和负载均衡配置.md
```

**优先级 (Priority)**:
```
作用: 控制渠道选择的优先顺序
取值: 0-999999 (数字越大越优先)

工作原理:
1. 先按优先级从高到低排序
2. 在同优先级内按权重随机选择
3. 当前优先级都失败才尝试下一层

示例配置:
核心渠道: 优先级 100 → 第1层,优先选择
备用渠道: 优先级 50  → 第2层,核心失败时选择
应急渠道: 优先级 0   → 第3层,最后的保障

使用场景:
- VIP渠道优先
- 官方优先,中转备用
- 快速优先,慢速兜底

详见: 03-容灾和负载均衡配置.md
```

**标签 (Tag)**:
```
作用: 给渠道打标签,方便分类和筛选
用途:
- 日志筛选和分析
- 渠道管理和组织
- 团队协作标识

示例:
- "主力": 主要使用的渠道
- "备用": 备用渠道
- "测试": 测试专用
- "美国": 美国地区IP
- "便宜": 成本较低的渠道

可以组合使用: "主力,美国,快速"
```

**备注 (Remark)**:
```
作用: 记录渠道相关的说明信息
建议记录:
- 购买渠道和时间
- 账号信息 (不要记录密码!)
- 特殊配置说明
- 维护记录
- 到期时间

示例:
"2025-01购买,年付$100,2026-01到期"
"代理IP: 123.45.67.89,美国洛杉矶"
"备用渠道,仅故障时启用"

好处:
- 方便团队协作
- 记录重要信息
- 到期续费提醒
```

### 5.2 常见错误和解决方法

#### 错误1: "401 Unauthorized"

**含义**: API密钥无效或已过期

**解决步骤**:
```
1. 登录上游平台,确认密钥是否正确
2. 检查密钥是否过期
3. 尝试重新生成密钥
4. 在New API中更新密钥
5. 重新测试
```

#### 错误2: "404 Not Found"

**含义**: API地址错误或路径不对

**解决步骤**:
```
1. 检查"代理"字段的URL是否正确
2. 确认URL格式:
   ✅ https://api.example.com
   ✅ https://api.example.com/v1
   ❌ https://api.example.com/ (多余的/)
   ❌ api.example.com (缺少https://)
3. 查看上游文档确认正确地址
4. 更新后重新测试
```

#### 错误3: "timeout" (超时)

**可能原因**:
- 网络问题
- 上游服务慢
- 需要代理但未配置
- 代理配置错误

**解决步骤**:
```
1. 检查服务器网络连接
   ping api.upstream.com
2. 如果是境外服务,配置代理
3. 检查代理配置格式是否正确
4. 增加超时时间 (设置→运营设置→超时设置)
```

#### 错误4: "The model 'xxx' does not exist"

**含义**: 上游不支持该模型

**解决步骤**:
```
1. 查看上游文档,确认支持的模型列表
2. 编辑渠道 → 更新"模型"字段
3. 选择上游支持的模型进行测试
4. 更新用户文档中的可用模型列表
```

#### 错误5: "Insufficient balance" (余额不足)

**含义**: 上游账户余额不足

**解决步骤**:
```
1. 登录上游平台查看余额
2. 给上游账户充值
3. 或者暂时禁用该渠道
4. 启用其他有余额的渠道
5. 设置余额告警 (避免再次发生)
```

### 6.1 渠道数量和组合建议

推荐配置: 2-3个渠道

预算分配:

```
推荐配置: 5-8个渠道
    主力渠道 × 3 (权重15-20)

预算分配:
    主力渠道: $200-500
    备选渠道: $100-200

推荐配置: 10+ 个渠道
    顶级渠道 × 3-5 (权重20)
    主力渠道 × 3-5 (权重12-15)
    备用渠道 × 3-5 (权重5-10)

策略:
- 多供应商分散风险
- 按地区分组 (国内、国外)
- 按性能分级 (快速、慢速、备用)
```

### 6.2 供应商选择标准

| 维度 | 权重 | 评分标准 |
|------|------|----------|

| 响应速度 | 25% | <1500ms = 满分 |
| 价格 | 20% | 比官方便宜20%+ = 满分 |
| 模型丰富度 | 15% | 支持10+模型 = 满分 |
| 客服支持 | 10% | 响应快、服务好 = 满分 |

**选择建议**:
```
✅ 优先选择:
- 知名度高的平台 (口碑好,不易跑路)
- 运营时间长 (>1年)
- 提供工单/客服支持
- 有退款保障

❌ 谨慎选择:
- 价格过低 (可能不稳定)
- 新平台 (风险较高)
- 无客服或联系方式
- 网站做工粗糙
```

### 6.3 成本控制策略

**策略1: 分级定价**

```
针对不同渠道成本,设置不同倍率:

便宜渠道:
    成本: 官方价 × 0.8
    倍率设置: 1.5
    利润率: (1.5 - 0.8) / 0.8 = 87.5%

正常渠道:
    成本: 官方价 × 1.0
    倍率设置: 1.5
    利润率: (1.5 - 1.0) / 1.0 = 50%

昂贵渠道:
    成本: 官方价 × 1.2
    倍率设置: 1.8
    利润率: (1.8 - 1.2) / 1.2 = 50%

结论: 通过分级定价,保持利润率一致
```

**策略2: 峰谷调度**

```
高峰期 (白天):
    启用快速但稍贵的渠道
    权重提高

低谷期 (夜间):
    启用便宜但可能慢的渠道
    权重提高

实现:
    手动调整权重
    或通过定时脚本自动化
```

**策略3: 流量导向**

```
低价值流量 (如gpt-3.5-turbo):
    使用便宜渠道
    分组: budget

高价值流量 (如gpt-4):
    使用稳定优质渠道
    分组: premium

实现:
    为不同模型的令牌指定不同分组
```

### 6.4 渠道命名规范

**推荐命名格式**:

```
[供应商]-[特征]-[序号]

示例:
API2D-主力-01
AIProxy-备用-02
CloseAI-快速-03
XXX-便宜-04
测试-内部-99
```

**特征标签建议**:
- 性能: 快速、慢速、普通
- 用途: 主力、备用、测试、VIP
- 地区: 国内、国外、美国、欧洲
- 成本: 便宜、正常、昂贵

**好处**:
- 一目了然的渠道用途
- 方便日志分析
- 便于团队协作

## 📊 七、渠道监控和数据分析

### 7.1 通过日志分析渠道表现

**步骤1: 按渠道统计使用情况**

```
路径: 日志 → 高级筛选

操作:
1. 时间范围: 选择"最近7天"
2. 导出CSV或在页面查看
3. 使用Excel按"渠道名"分组统计

数据指标:
- 请求总数
- 成功数 / 失败数
- 成功率 = 成功数 / 请求总数
- 平均响应时间
- 总消耗金额
```

**步骤2: 识别问题渠道**

```
筛选条件:
⚠️ 成功率 < 95%
⚠️ 平均响应时间 > 5000ms
⚠️ 失败次数 > 100次/天

处理动作:
1. 降低权重
2. 联系上游排查
3. 如持续问题,禁用渠道
```

**步骤3: 识别优质渠道**

```
筛选条件:
✅ 成功率 > 99%
✅ 平均响应时间 < 2000ms
✅ 稳定运行(无大幅波动)

奖励动作:
1. 提高权重
2. 优先分配高价值请求
3. 增加该供应商的渠道数量
```

### 7.2 设置告警监控

**推荐监控指标**:

```
渠道级别:
- 成功率 < 90% (过去1小时)
- 响应时间 > 10秒 (单次)
- 连续失败 > 10次
- 余额 < $10

系统级别:
- 所有渠道失败率 > 50%
- 单小时请求量暴增 (>正常3倍)
- 单用户消费异常 (>$100/小时)
```

**告警渠道**:
- 邮件通知
- 企业微信/钉钉机器人
- Telegram Bot
- 短信 (重要告警)

### 7.3 性能优化建议

**优化1: 禁用慢速渠道**

```
标准: 响应时间持续 > 8000ms

操作:
渠道 → 编辑 → 状态改为"已禁用"

或调整权重为1-2 (降低流量)
```

**优化2: 按地域优化**

```
如果用户主要在国内:
    优先使用国内渠道 (响应更快)
    国外渠道作为备份

如果用户主要在国外:
    使用国外渠道
    配置CDN加速
```

**优化3: 按模型分组**

```
创建模型专用渠道:

分组: gpt4-专用
    └─ 仅添加GPT-4稳定的渠道
    └─ 为使用GPT-4的令牌指定此分组

分组: 通用
    └─ 添加所有模型
    └─ 普通令牌使用
```

## ✅ 总结

### 核心要点回顾

1. **渠道 = 上游API连接**: 一个账号一个渠道
2. **最常见类型: OpenAI**: 90%的中转站点选这个
3. **必填字段**: 类型、名称、密钥、代理(Base URL)、模型
4. **添加后先测试**: 确保配置正确
5. **建议2+渠道**: 互为备份,提高可用性
6. **权重控制流量**: 权重越高,分配流量越多
7. **定期检查维护**: 每日状态,每周测试,每月优化

### 常见错误避免

- ❌ API地址填错 (多余的/,缺少https://)
- ❌ 密钥复制不完整
- ❌ 模型列表填错 (上游不支持)
- ❌ 不配置代理 (导致被封号)
- ❌ 只用一个渠道 (单点故障风险)
- ❌ 不监控余额 (突然欠费中断服务)
