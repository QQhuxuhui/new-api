### 2.2 健康检查机制

**位置**: `service/channel_health.go`

#### 核心参数

```go
// 滑动窗口参数
WindowDuration        = 60 * time.Second  // 窗口时长
BucketSize           = 10 * time.Second  // 分桶粒度
BucketCount          = 6                 // 6个桶

// 失败率阈值
FailureRateThreshold     = 0.30  // 标准流量：30%
FailureRateThresholdHigh = 0.50  // 低流量：50%
MinSampleSize            = 5     // 最小样本数
LowTrafficThreshold      = 30    // 低流量门槛（请求/分钟）
LowTrafficMinFailures    = 5     // 低流量最少失败数
LowTrafficFailureRate    = 0.80  // 低流量：80%

// 状态阈值
SuspensionThreshold      = 3       // 3个高失败周期→暂停
DisableThreshold         = 10      // 10个周期→触发永久禁用（未实现，仅打印日志）
FixedSuspensionDuration  = 2分钟   // 固定暂停时长
```

> **重要说明**: `DisableThreshold` 的永久禁用功能**当前未实现**。当连续高失败率周期达到10次时，`disableChannelPermanently()` 函数仅打印日志，不会真正禁用渠道。实际的渠道禁用由错误驱动机制（`ShouldDisableChannel`）处理。

#### 失败率判定逻辑

**函数**: `IsHighFailureRate()` (行191-232)

```
判定流程：
1. 低流量快速识别 (2样本+80%失败率) → 立即标记
2. 低流量阈值检查 (5失败+80%失败率) → 标记
3. 正常流量检查：
   - 请求量 < 30/分钟 → 失败率 > 50% 触发
   - 请求量 >= 30/分钟 → 失败率 > 30% 触发
```

#### 健康状态结构

```go
type ChannelHealth struct {

    SuspensionCount     int       // 暂停次数（指数退避）
```

### 2.5 暂停与恢复机制

#### 暂停机制（固定时长）

```
暂停时间：固定 2 分钟

设计原因：
- 上游恢复时间不确定，较短暂停时间可更快探测恢复状态
- 滑动窗口机制会在暂停到期后重新评估，如仍故障则再次暂停
- 避免指数退避导致长时间错过可用时段
```

> **注意**: 当连续高失败率周期达到10次时，会触发 `disableChannelPermanently()` 函数，但该函数当前仅打印日志，实际的永久禁用由 `ShouldDisableChannel()` 的错误驱动禁用机制处理。

#### 恢复机制

**函数**: `RecordChannelSuccess()`

```go
成功请求时：
1. 记录成功到滑动窗口
2. 重置连续失败计数
3. 移除暂停标志
4. 重置暂停计数
5. 更新最后成功时间
6. 增加总成功计数
```

## 三、渠道选择策略

### 3.1 选择流程入口

```
┌─────────────────────────────────────────────────────────────┐
│  middleware/distributor.go - Distribute()                   │
│  ↓                                                          │
│  service/channel_select.go - CacheGetRandomSatisfiedChannel │
│  ↓                                                          │
│  model/channel_cache.go - GetRandomSatisfiedChannel         │
└─────────────────────────────────────────────────────────────┘
```

### 3.2 优先级策略

**位置**: `model/channel_cache.go:170-188`

```
工作原理：
1. 收集所有可用渠道的优先级值
2. 去重并降序排列
3. retry参数指定尝试的优先级级别：
   - retry=0 → 最高优先级
   - retry=1 → 次高优先级
   - retry=N → 第(N+1)高优先级
4. retry >= 优先级数量 → ErrPriorityExhausted
```

### 3.3 权重策略

**位置**: `model/channel_cache.go:219-245`

```
同一优先级内的负载均衡：

1. 计算该优先级所有健康渠道的总权重
2. 权重平滑处理：
   - 总权重=0 → 每渠道有效权重=100
   - 平均权重<10 → 放大因子=100
3. 生成 [0, totalWeight) 随机数
4. 遍历渠道，累减权重
5. 第一个使随机数变负的渠道被选中
```

### 3.4 组选择策略

**位置**: `service/channel_select.go:14-102`

```
优先级顺序：
1. 计划组 (Plan Groups)
   - 有计划 → 使用计划指定的渠道组列表

2. 自动模式 (group="auto")
   - 遍历用户所有可用组
   - 按优先级顺序选择

3. 特定组 (Specific Group)
   - 使用用户指定的单一组
```

## 四、渠道切换与降级机制

### 4.1 多层级降级架构

```
┌─────────────────────────────────────────────────────────────┐
│  第一层：优先级降级 (Priority Degradation)                   │
│  retry=0 → retry=1 → ... → ErrPriorityExhausted             │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│  第二层：计划降级 (Plan Failover)                            │
│  当前计划耗尽 → 查找其他有渠道的计划                          │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│  第三层：钱包回退 (Wallet Fallback)                          │
│  所有计划耗尽 → 使用钱包余额消费                              │
└─────────────────────────────────────────────────────────────┘
```

### 4.2 优先级重试循环

**位置**: `middleware/distributor.go:437-500`

```go
const maxPriorityLevels = 1000

for retry := 0; retry < maxPriorityLevels; retry++ {
    channel, selectGroup, err = CacheGetRandomSatisfiedChannel(...)

    if errors.Is(err, model.ErrPriorityExhausted) {
        break  // 所有优先级都已尝试
    }

    if channel != nil {
        break  // 找到健康渠道
    }
    // channel == nil 表示该优先级无健康渠道，继续下一个
}
```

### 4.3 计划降级实现

**位置**: `service/plan_failover.go:79-242`

```
AttemptPlanFailover() 流程：

1. 获取故障转移候选计划
   - 排除当前计划
   - 检查权限、配额

2. 按优先级遍历候选计划的渠道组

3. 对每个组执行完整的优先级重试

4. 返回第一个找到的健康渠道和对应计划
```

### 4.4 跨计划降级

**位置**: `controller/relay.go:224-313`

触发条件：
- 计划系统启用
- 用户有有效的 current_plan
- 计划设置了 `auto_switch=1`

执行流程：
```
1. 返还预消耗配额
2. 查找替代计划及其健康渠道
3. 更新计费源和配额信息
4. 重新计算价格数据
5. 预消耗新计划配额
6. 使用新渠道重试请求
```

### 4.5 钱包降级

**位置**: `middleware/distributor.go:609-710`

```go
if channel == nil {
    // 检查是否为父组
    if ratio_setting.IsParentGroup(tokenGroup) {
        // 找出未尝试过的子组
        untriedGroups := getUntriedGroups(allChildGroups, triedGroups)

        // 尝试未尝试的子组
        if len(untriedGroups) > 0 && userQuota > 0 {
            for _, childGroup := range untriedGroups {
                channel, _, err = CacheGetRandomSatisfiedChannel(...)
                if channel != nil {
                    // 切换到钱包消费模式
                    c.Set("plan_id", 0)
                    break
                }
            }
        }
    }
}
```

## 五、重试逻辑

### 5.1 控制层重试

**位置**: `controller/relay.go:160-222`

```go
for i := 0; i <= common.RetryTimes; i++ {
    channel, err := getChannel(c, group, originalModel, i)

    if types.IsSkipRetryError(err) {
        break  // 跳过重试
    }

    newAPIError = relayHandler(c, relayInfo)

    if newAPIError == nil {
        service.RecordChannelSuccess(channel.Id)
        return  // 成功
    }

    if !shouldRetry(c, newAPIError, common.RetryTimes-i) {
        break  // 不应重试
    }
}
```

### 5.2 重试判定逻辑

**函数**: `shouldRetry()` (relay.go:367-404)

| 条件 | 动作 |
|------|------|
| SkipRetryError | 不重试 |
| IsChannelError | 重试（绕过RetryTimes限制）|
| 指定特定渠道 | 不重试 |
| 429 Too Many Requests | 重试 |
| 307 重定向 | 重试 |
| 5xx（除504/524）| 重试 |
| 504/524 超时 | 仅首次失败时重试1次 |
| RetryTimes耗尽 | 不重试 |

### 5.3 并发限制重试

**位置**: `middleware/distributor.go:750-835`

当渠道达到最大并发时，会执行以下逻辑：

```go
if err.GetErrorCode() == types.ErrorCodeChannelKeyConcurrencyLimit {
    // 1. 在当前计划内尝试其他渠道
    for retry := 0; retry < maxRetryAttempts; retry++ {
        // 选择排除已尝试渠道的新渠道
        channel = CacheGetRandomSatisfiedChannelExcluding(..., triedChannelIds)
        if channel != nil {
            triedChannelIds[channel.Id] = true  // 记录
        }
    }

    // 2. 所有渠道都达到并发限制时，触发跨计划降级
    if newAPIError.GetErrorCode() == types.ErrorCodeChannelKeyConcurrencyLimit {
        // 检查是否满足降级条件（用户有其他计划、计划启用auto_switch）
        if shouldFailover, currentUserPlanId := shouldAttemptPlanFailover(...); shouldFailover {
            failoverChannel, failoverPlan, success := executePlanFailover(...)
            if success {
                // 使用新计划的渠道
                channel = failoverChannel
            }
        }
    }
}
```

#### 并发限制触发跨计划降级

**位置**: `middleware/distributor.go:814-835`

触发条件：
- 当前计划的所有渠道都达到并发限制
- 用户有其他有效计划
- 当前计划启用了 `auto_switch`
- 不是指定特定渠道的请求

执行流程：
1. 调用 `executePlanFailover()` 查找替代计划
2. 检查替代计划的速率限制
3. 设置替代渠道的上下文
4. 使用新渠道继续请求

## 六、立即故障转移判定

**位置**: `service/error.go:62-88`

**函数**: `ShouldImmediateFailover()`

```
立即故障转移的条件：

1. 403 + 并发/资源耗尽关键词
   - "并发", "session" + "已满", "concurrency", "overloaded"

2. 401 + 认证问题关键词
   - "invalid", "expired", "authentication"

3. 配额耗尽（任何状态码）
   - "insufficient_quota"
   - "quota exceeded"
   - "billing_not_active"
```

**函数**: `ShouldTriggerChannelFailover()` (行90-139)

设计原则：基于HTTP状态码判断，不依赖错误消息关键词匹配

| 状态码范围 | 行为 | 说明 |
|-----------|------|------|
| 2xx | 不触发 | 成功响应 |
| 400 | 不触发 | 客户端请求格式问题 |
| 401, 403, 404-428, 430-499 | 触发 | 认证/权限/客户端错误 |
| 429 | 触发 | 速率限制 |
| 5xx（除504/524）| 触发 | 服务器错误 |
| 504/524 | 不触发（单独处理）| 超时在调用处处理 |
| 网络错误关键词 | 触发 | connection/timeout/dns/tls/ssl/network |

### 7.2 Key选择逻辑

**函数**: `GetNextEnabledKey()`

```
1. 收集所有启用的Key索引
2. 根据 MultiKeyMode 选择：
   - random: 随机选择
   - polling: 从上次位置轮询
3. 更新轮询索引
4. 所有Key禁用 → 返回错误
```

### 7.3 整渠道禁用判定

```go
// 当某个Key被禁用时
handlerMultiKeyUpdate() {

    }
}
```

```
    │
    ▼
┌─────────────────────────────────────────────────────────────┐
│  计划降级                                                     │
│  ├─ 查找替代计划（排除当前，检查权限/配额）                    │

│  └─ 未找到 → 钱包回退                                         │
└─────────────────────────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────────────────────────┐
│  Relay 控制层重试 (i=0..RetryTimes)                          │
│  ├─ 预消耗配额                                                │
│  ├─ 执行API请求                                               │
│  ├─ 成功 → RecordChannelSuccess                              │
│  ├─ 失败 → shouldRetry判定                                   │
│  │   ├─ SkipRetry → 中止                                     │
│  │   ├─ 渠道错误 → 重试                                       │

│  │   └─ 其他 → 中止                                          │
│  └─ 所有重试失败 → 跨计划降级 (Controller层)                   │
└─────────────────────────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────────────────────────┐
│  健康系统（并行运行）                                         │
│  ├─ 记录请求结果到滑动窗口                                     │
│  ├─ 计算失败率（每60秒）                                       │
│  ├─ 失败率>阈值 → 连续失败+1                                  │
│  ├─ 连续失败>=3 → 暂停渠道（指数退避）                         │
│  └─ 连续失败>=10 → 触发永久禁用日志（实际禁用由错误驱动）       │
└─────────────────────────────────────────────────────────────┘
```

## 十二、可配置故障转移规则（新增）

为提升对上游异常的适配度，支持通过「渠道故障转移规则」页面（运营设置内）配置自定义匹配条件：

- 四种匹配模式：AND、OR、STATUS_ONLY、KEYWORD_ONLY
- 支持状态码列表与错误关键词（不区分大小写）
- 匹配成功会记录到健康检查滑动窗口，触发临时暂停，**不会**永久禁用渠道
- 规则与硬编码逻辑按 OR 关系叠加，硬编码命中会短路返回
- 提供测试面板，可查看硬编码命中情况、各规则匹配详情、最终是否触发

API 路径：
- `GET /api/channel/disable-rules` 列表
- `POST /api/channel/disable-rules` 新建
- `PUT /api/channel/disable-rules/:id` 更新
- `DELETE /api/channel/disable-rules/:id` 删除
- `POST /api/channel/disable-rules/test` 测试匹配

缓存策略：
- 启用规则缓存 5 分钟
- CRUD 操作后立即失效缓存
- 过期或失效后按优先级降序刷新最新规则

## 总结

本系统实现了**多层次、多策略**的渠道可用性管理：

1. **实时健康监控** - 60秒滑动窗口，精确的失败率计算
2. **自动暂停恢复** - 指数退避暂停，成功请求自动恢复
3. **错误驱动禁用** - 基于错误码、错误类型、关键词的智能禁用
4. **多级降级机制** - 优先级降级 → 计划降级 → 钱包回退
5. **智能重试策略** - 根据错误类型决定是否重试及重试次数
6. **并发限制降级** - 所有渠道达到并发限制时触发跨计划降级

确保请求能够自动故障转移到健康的渠道，提供高可用性保障。
