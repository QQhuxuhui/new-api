# 健康检查机制遗漏分析与修复

## 修复状态
✅ **所有遗漏已修复** (2025-11-20)

---

## 问题来源

用户报告后台出现 504 错误日志：
```
[ERR] channel error (channel #4, status code: 504): bad response status code 504
[INFO] record error log: userId=4, channelId=4, content=bad response status code 504
[ERR] relay error: bad response status code 504
```

通过分析发现：虽然 `improve-channel-failover-detection` 提案已经实施，但仍存在多个遗漏导致健康检查机制不完整。

---

## 发现的遗漏

### 1. Relay: 504 错误不记录到健康检查 ✅ 已修复

**影响范围**: 常规 Relay 路径（OpenAI, Claude, Gemini 等）

**问题**:
- 504 Gateway Timeout 不触发重试（设计如此）
- 但也不被记录到健康检查系统（遗漏）
- 频繁 504 的通道不会被自动暂停

**修复**: `controller/relay.go:196-203`
```go
// 修复前
if service.ShouldTriggerChannelFailover(newAPIError.StatusCode, newAPIError.Error()) {
    service.RecordChannelFailure(...)
}

// 修复后
if service.ShouldTriggerChannelFailover(newAPIError.StatusCode, newAPIError.Error()) ||
    newAPIError.StatusCode == 504 || newAPIError.StatusCode == 524 {
    service.RecordChannelFailure(...)  // 记录但不立即重试
}
```

**效果**:
- ✅ 504 不触发立即重试（避免雪崩）
- ✅ 504 记录到健康检查（累积统计）
- ✅ 失败率 >30% 自动暂停通道

### 2. RelayTask: 完全没有健康检查记录 ✅ 已修复

**影响范围**:
- Suno 音乐生成
- Video 视频生成
- 所有异步任务型请求

**问题**:
- `RelayTask()` 没有调用任何健康检查函数
- 任务型请求的通道健康状态不被追踪
- 故障通道持续接收请求，用户持续失败

**修复**: `controller/relay.go:424-480`
```go
// 第一次尝试后记录
taskErr := taskRelayHandler(c, relayInfo)
if taskErr == nil {
    service.RecordChannelSuccess(channelId)  // ✅ 新增
    retryTimes = 0
} else {
    // ✅ 新增健康检查记录
    if service.ShouldTriggerChannelFailover(taskErr.StatusCode, taskErr.Message) ||
        taskErr.StatusCode == 504 || taskErr.StatusCode == 524 {
        service.RecordChannelFailure(channelId, taskErr.StatusCode, taskErr.Message)
    }
}

// 重试循环中也记录
for i := 0; shouldRetryTaskRelay(...) && i < retryTimes; i++ {
    taskErr = taskRelayHandler(c, relayInfo)
    if taskErr == nil {
        service.RecordChannelSuccess(channelId)  // ✅ 新增
        return
    }
    // ✅ 新增健康检查记录
    if service.ShouldTriggerChannelFailover(...) || statusCode == 504 || statusCode == 524 {
        service.RecordChannelFailure(...)
    }
}
```

**效果**:
- ✅ Suno/Video 请求失败会被记录
- ✅ 故障通道自动暂停
- ✅ 自动切换到健康通道

### 3. RelayMidjourney: 完全没有健康检查记录 ✅ 已修复

**影响范围**:
- Midjourney 图像生成
- SwapFace 换脸功能

**问题**:
- `RelayMidjourney()` 没有健康检查记录
- 没有重试逻辑（单次失败直接返回）
- 故障通道不被标记

**修复**: `controller/relay.go:357-409`
```go
channelId := c.GetInt("channel_id")
// ... 执行请求 ...

if mjErr != nil {
    statusCode := http.StatusBadRequest
    if mjErr.Code == 30 {
        statusCode = http.StatusTooManyRequests
    }

    // ✅ 新增：记录失败
    errorMessage := fmt.Sprintf("%s %s", mjErr.Description, mjErr.Result)
    if statusCode >= 500 || statusCode == http.StatusTooManyRequests ||
       statusCode == 504 || statusCode == 524 {
        service.RecordChannelFailure(channelId, statusCode, errorMessage)
    }

    c.JSON(statusCode, ...)
} else {
    // ✅ 新增：记录成功
    service.RecordChannelSuccess(channelId)
}
```

**效果**:
- ✅ Midjourney 请求失败会被记录
- ✅ 故障通道自动暂停
- ✅ 健康检查覆盖所有请求类型

**注意**: Midjourney 暂未添加重试逻辑，需评估 API 幂等性后再决定。

## 为什么会遗漏？

### 1. 提案聚焦主流程
- `improve-channel-failover-detection` 提案主要关注常规 Relay 路径
- 没有明确要求覆盖 Task 和 Midjourney 路径

### 2. 代码结构分散
- 三个独立入口：`Relay()`, `RelayTask()`, `RelayMidjourney()`
- 没有统一的健康检查抽象层
- 容易漏掉某些路径

### 3. 健康检查是后加功能
- 健康检查系统是后期添加的
- 只在主流程 `Relay()` 中集成
- 其他路径没有同步更新

### 4. 概念混淆
- **不触发立即重试**: 单个请求不切换通道（避免雪崩）
- **记录到健康检查**: 累积统计，频繁失败自动暂停
- 提案说 "504 不重试"，但没说 "504 不记录"，导致误解
