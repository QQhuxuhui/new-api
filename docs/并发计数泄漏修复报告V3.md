# 并发计数泄漏问题最终修复报告（V3）

## 🚨 V2 版本的严重缺陷

V2 版本虽然引入了时间戳机制，但存在**三个严重问题**，会导致**并发限制完全失效**：

### [严重] 问题 1: 时间戳只在首次设置，正常请求被误删

**缺陷位置：** `service/concurrency.go:82-84`

```go
// V2 错误代码
pipe.SetNX(ctx, timestampKey, time.Now().Unix(), 1*time.Hour)
```

**问题：**
- 使用 `SetNX` 只在第一次设置时间戳
- 后续 Incr 不更新时间戳
- 长时间运行的请求（如流式调用）或持续有请求的 key，时间戳一直不变
- 清理任务检查"存在超过 2 分钟"，误判为泄漏并删除

**影响：**
- ❌ 正在使用的并发计数被删除
- ❌ 新请求重新从 0 计数
- ❌ 旧请求结束时 Decr 一个不存在的 key（得到 -1）
- ❌ **并发限制完全失效，可能突破 MaxConcurrentRequestsPerKey**

### [严重] 问题 2: 时间戳残留导致死循环

**缺陷位置：** `service/concurrency_cleanup.go:93-96`

```go
// V2 错误代码
if err != nil || count <= 0 {
    common.RDB.Del(ctx, key)  // ❌ 只删除主 key，时间戳残留
    continue
}
```

**问题：**
- 删除 count <= 0 的 key 时，没有删除 :timestamp
- 下次合法请求到来时，`SetNX` 因旧时间戳存在返回 false
- 时间戳停留在旧值，2 分钟后新计数又被误删

**影响：**
- ❌ 形成"创建→误删→重新创建"的死循环
- ❌ 并发限制极易被绕过

### [中等] 问题 3: 清理间隔与阈值不匹配

**缺陷位置：** `service/concurrency_cleanup.go:34-35`

```go
// V2 错误代码
// 注释：每 5 分钟执行一次清理（比泄漏阈值更频繁）
ticker := time.NewTicker(5 * time.Minute)  // 但阈值是 2 分钟！
```

**问题：**
- 阈值 2 分钟，但清理间隔 5 分钟
- 注释说"比泄漏阈值更频繁"，实际恰恰相反
- 泄漏至少要等 5 分钟才会被处理

**影响：**
- ⚠️ 清理延迟过长
- ⚠️ 注释与实际行为不符，误导排障

## ✅ V3 最终修复方案

### 修复 1: 时间戳改为"最后活动时间"

**文件：** `service/concurrency.go:76-87`

```go
// V3 正确代码
// Increment counter and update timestamp using pipeline
_, err = tx.TxPipelined(ctx, func(pipe redis.Pipeliner) error {
    pipe.Incr(ctx, redisKey)
    pipe.Expire(ctx, redisKey, 1*time.Hour)

    // ✅ 使用 Set 而不是 SetNX，每次都更新时间戳
    // 这样时间戳反映的是"最后活动时间"而不是"首次创建时间"
    pipe.Set(ctx, timestampKey, time.Now().Unix(), 1*time.Hour)
    return nil
})
```

**修复效果：**
- ✅ 每次 Incr 都更新时间戳为当前时间
- ✅ 长时间运行的请求不会被误判为泄漏
- ✅ 持续有活动的 key 不会被清理

### 修复 2: 删除 key 时同步删除时间戳

**文件：** `service/concurrency_cleanup.go:92-98`

```go
// V3 正确代码
count, err := strconv.Atoi(val)
if err != nil || count <= 0 {
    // ✅ 删除主 key 和时间戳 key
    timestampKey := GetConcurrencyTimestampKey(key)
    common.RDB.Del(ctx, key, timestampKey)
    continue
}
```

**修复效果：**
- ✅ 删除计数时同步删除时间戳
- ✅ 下次请求能正确设置新时间戳
- ✅ 避免死循环

### 修复 3: 清理间隔自适应阈值

**文件：** `service/concurrency_cleanup.go:27-52`

```go
// V3 正确代码
// ✅ 清理间隔设为泄漏阈值的一半
cleanupInterval := ConcurrencyLeakThreshold / 2
if cleanupInterval < 30*time.Second {
    cleanupInterval = 30 * time.Second // 最小间隔 30 秒
}

ticker := time.NewTicker(cleanupInterval)
```

**修复效果：**
- ✅ 阈值 2 分钟 → 清理间隔 1 分钟
- ✅ 阈值 5 分钟 → 清理间隔 2.5 分钟
- ✅ 自适应，确保及时发现泄漏

### 修复 4: 处理旧版本 key

**文件：** `service/concurrency_cleanup.go:112-119`

```go
// V3 正确代码
if err == redis.Nil {
    // ✅ 时间戳不存在，无法判断最后活动时间，直接删除
    common.SysLog(fmt.Sprintf("Cleaned concurrency key without timestamp (legacy or corrupted): key=%s, count=%d", key, count))
    common.RDB.Del(ctx, key, timestampKey)
    cleanedCount++
    continue
}
```

**修复效果：**
- ✅ 旧版本 key 被自动清理
- ✅ 避免无时间戳的 key 永久存在

## 🔍 工作原理

### 时间戳语义变化

```
V2: 时间戳 = 首次创建时间（SetNX）
问题：长请求被误判为"存在太久"

V3: 时间戳 = 最后活动时间（Set）
效果：只要有活动就更新，不会被误删
```

### 清理逻辑

```
V3 清理条件：
1. 时间戳存在 && (当前时间 - 最后活动时间) > 2 分钟  → 泄漏，删除
2. 时间戳不存在                                    → 旧 key，删除
3. count <= 0                                      → 无效，删除（含时间戳）
```

### 时间线示例

```
阈值 = 2 分钟，清理间隔 = 1 分钟

0:00 - 请求开始，Incr，时间戳 = 0:00
0:30 - 另一个请求，Incr，时间戳 = 0:30 (更新！)
1:00 - 清理扫描，age = 0.5 分钟 < 2 分钟，跳过 ✅
1:30 - 又一个请求，Incr，时间戳 = 1:30 (更新！)
2:00 - 清理扫描，age = 0.5 分钟 < 2 分钟，跳过 ✅
...
5:00 - 最后一个请求完成，Decr，时间戳 = 5:00
5:01 - 如果服务崩溃，没有 Decr
7:01 - 清理扫描，age = 2.01 分钟 > 2 分钟，删除 ✅
```

### ⚠️ 长时间请求配置注意事项

**重要：阈值必须大于正常请求的最长耗时**

V3 的时间戳机制基于"最后活动时间"，但对于某些场景仍需注意配置：

#### 场景分析

1. **短请求场景（推荐默认配置）**
   - 典型请求耗时：< 60 秒
   - 配置：使用默认 `CONCURRENCY_LEAK_THRESHOLD=120` （2 分钟）
   - 风险：低 - 正常请求远短于阈值

2. **长请求场景（需调整配置）**
   - 流式调用：持续时间可能超过数分钟
   - 大文件传输：耗时可能 5-10 分钟
   - 长连接：保持活跃状态较长
   - **配置建议：**
     ```bash
     # 设置为"肯定大于正常请求最长耗时"的值
     # 例如：最长请求 8 分钟，设置为 10-15 分钟
     export CONCURRENCY_LEAK_THRESHOLD=600  # 10 分钟
     # 或
     export CONCURRENCY_LEAK_THRESHOLD=900  # 15 分钟
     ```

3. **混合场景**
   - 大部分短请求 + 少量长请求
   - 配置：按最长请求配置（宁可清理慢，不能误删）
   - 示例：`CONCURRENCY_LEAK_THRESHOLD=600` （10 分钟）

#### 配置原则

```
阈值 > 99% 请求的最长耗时 + 30秒缓冲
```

**示例计算：**
- 监控显示 99% 的请求在 5 分钟内完成
- 建议阈值：5 分钟 + 30 秒 = 330 秒
- 保守配置：`CONCURRENCY_LEAK_THRESHOLD=360` （6 分钟）

#### 验证方法

部署后观察日志：
```bash
# 检查是否有正常请求被误清理
tail -f logs/app.log | grep "Cleaned leaked concurrency"

# 如果看到频繁清理且业务反馈并发限制不准确，说明阈值过小
# 应增大 CONCURRENCY_LEAK_THRESHOLD
```

## ⚠️ 重要提醒

### 必须清理旧数据

V2 和 V3 的时间戳语义不同：
- **V2:** 首次创建时间（SetNX）
- **V3:** 最后活动时间（Set）

**如果不清理旧数据：**
- V2 留下的时间戳永远不会更新
- 所有 V2 的 key 会在 2 分钟后被 V3 误删
- 导致并发限制暂时失效

**解决方案：**
```bash
# 升级前必须执行
REDIS_CONN_STRING=redis://your-redis:6379 go run scripts/cleanup_concurrency.go
```
