📋 各环节详细工作内容

1
客户端发起请求
支持多种协议和 API 格式
  • 支持的协议
    • HTTP/HTTPS
    • WebSocket（实时 API）
  • 支持的 API 格式
    • OpenAI 格式：/v1/chat/completions
    • Claude 格式：/v1/messages
    • Gemini 格式：/v1beta/models/xxx:generateContent
    • 实时 API：/v1/realtime
    • Midjourney：/mj/submit/imagine
  • 请求携带信息
    • API Key（Authorization header 或其他方式）
    • 模型名称
    • 请求参数（prompt, messages, temperature...）
2
路由匹配
文件：router/relay-router.go
  • 路径匹配 根据请求路径匹配对应的处理器（Controller）
  • 识别 API 格式
    • OpenAI 格式 → relayV1Router
    • Claude 格式 → httpRouter
    • Gemini 格式 → relayGeminiRouter
  • 应用中间件链
    • CORS 中间件
    • 请求解压中间件
    • 统计中间件
    • 认证中间件（下一环节）
3
Token 认证（TokenAuth 中间件）
文件：middleware/auth.go
  • ① 提取 Token 支持多种提取方式：
    • Bearer Token（Authorization: Bearer sk-xxx）
    • WebSocket（Sec-WebSocket-Protocol）
    • Anthropic（x-api-key header）
    • Gemini（query parameter 或 x-goog-api-key）
  • ② 验证 Token 状态
    • 检查 Token 是否存在
    • 检查状态（1=启用, 2=耗尽, 3=过期）
    • 验证过期时间
    • 检查剩余配额（如果非无限配额）
  • ③ IP 白名单检查
    • 如果 Token 配置了 IP 白名单
    • 提取客户端 IP
    • 验证 IP 是否在白名单内（支持 CIDR）
  • ④ 客户端限制检查（新特性）
    🆕 Client Restriction
    • 提取请求的 User-Agent
    • 应用三种匹配模式：
      • 精确匹配：Claude-Code-CLI/1.0.0
      • 通配符：VSCode/*
      • 正则：regex:^Postman.*
    • 不匹配则拒绝请求（403 Forbidden）
  • ⑤ 粘性会话配置（新特性）
    🆕 Sticky Session
    • 如果 Token 启用粘性会话
    • 设置粘性会话标志到请求上下文
    • 记录 TTL（默认 3600 秒）
  • ⑥ 用户权限验证
    • 加载用户信息
    • 验证用户状态
    • 验证分组权限
  • ⑦ 设置请求上下文
    • 保存 Token ID、Key 到 Context
    • 保存用户 ID、分组到 Context
    • 保存粘性会话配置到 Context
4
模型限流（RateLimit 中间件）
文件：middleware/model-rate-limit.go
  • ① 提取限流维度
    • Token ID
    • 模型名称
  • ② 检查限流配置
    • 查询该 Token + 模型的限流规则
    • 获取限流阈值（QPM、QPD 等）
  • ③ 执行限流判断
    • 使用 Token Bucket 或 Leaky Bucket 算法
    • 从 Redis 获取当前计数
    • 判断是否超过阈值
    • 超过 → 返回 429 Too Many Requests
    • 未超过 → 增加计数，放行
5
渠道分配（Distribute 中间件）
文件：middleware/distributor.go
  • ① 解析请求获取模型名称
    • 从请求体中提取 model 字段
    • 支持多种路径格式解析
  • ② Token 模型权限验证
    • 如果 Token 启用了 ModelLimitsEnabled
    • 检查请求的模型是否在 ModelLimits 列表中
    • 不在列表 → 拒绝请求（403 Forbidden）
  • ③ 渠道选择策略
    🔄 粘性会话模式（Sticky Session）
    • 获取会话用户 ID（userId 或 sessionId）
    • 查询 Redis：sticky_session:{userId}:{model}:{group}
    • 如果已绑定渠道：
      • 获取绑定的渠道 ID
      • 检查渠道健康状态（Status == Enabled）
      • 健康 ✅ → 使用绑定渠道，更新 TTL
      • 失效 ❌ → 解绑渠道，重新选择
    • 如果未绑定渠道：
      • 调用渠道选择算法（下方）
      • 绑定新渠道到会话
      • 设置 TTL（默认 3600 秒）
    🎲 普通模式（Random Selection）
    • 调用 CacheGetRandomSatisfiedChannel(group, model, 0)
    • 使用加权随机算法选择渠道
  • ④ 渠道选择算法（核心）
    文件：model/channel_cache.go::GetRandomSatisfiedChannel()
    • 步骤1：从缓存获取可用渠道列表
      channels := group2model2channels[group][model]
    • 步骤2：按优先级分组
      提取所有唯一的优先级，降序排序
    • 步骤3：根据 retry 次数选择优先级
      targetPriority = sortedPriorities[min(retry, len-1)]
      首次请求 → 最高优先级；重试 → 降级到低优先级
    • 步骤4：筛选目标优先级的渠道
    • 步骤5：权重归一化
      • 如果所有权重为 0 → 每个渠道权重 = 100
      • 如果平均权重 < 10 → 应用平滑因子 100
    • 步骤6：加权随机选择
      计算总权重，生成随机数，按权重分段选择渠道
  • ⑤ 获取渠道 Key（多 Key 管理）
    文件：model/channel.go::GetNextEnabledKey()
    • 如果渠道配置了多个 Key：
      • Random 模式：从可用 Key 中随机选择
      • Polling 模式：从上次轮询位置开始，顺序选择下一个可用 Key
    • 使用渠道级别的锁（channelPollingLocks）保证并发安全
    • 返回选中的 Key 和 Key 索引
  • ⑥ 设置渠道上下文
    • 保存渠道 ID、Type、Name
    • 保存渠道 Key、BaseURL
    • 保存模型映射、参数覆盖配置
    • 保存是否多 Key、Key 索引
6
转发控制器（Relay Controller）
文件：controller/relay.go
  • ① 请求验证和解析
    • 验证请求格式（JSON、Content-Type）
    • 解析请求体
    • WebSocket 升级（如果是实时 API）
  • ② 生成转发信息（RelayInfo）
    • 收集所有上下文信息（Token、渠道、用户...）
    • 创建 RelayInfo 对象
  • ③ 敏感词检测（如果启用）
    • 提取 prompt 或 messages
    • 调用敏感词检测服务
    • 包含敏感词 → 拒绝请求
  • ④ Token 计数
    • 使用 tiktoken 或其他工具计算 prompt token 数
    • 保存到 relayInfo.PromptTokens
  • ⑤ 价格计算
    • 根据模型和 token 数计算价格
    • 计算需要预扣的配额
    • 保存到 relayInfo.PriceData
  • ⑥ 预扣费
    • 从 Token 配额中预扣
    • 从用户配额中预扣
    • 记录预扣金额
  • ⑦ 重试循环
    for retry := 0; retry <= RetryTimes; retry++
    • 首次请求（retry=0）：使用 Distribute 选择的渠道
    • 重试请求（retry>0）：重新调用 CacheGetRandomSatisfiedChannel(group, model, retry)
      → 传入 retry 参数，降级到低优先级渠道
    • 调用适配器处理：
      • OpenAI 格式 → relay.TextHelper()
      • Claude 格式 → relay.ClaudeHelper()
      • Gemini 格式 → relay.GeminiHelper()
      • WebSocket → relay.WssHelper()
    • 成功 → 返回
    • 失败 → 错误处理：
      • 记录错误日志
      • 判断是否自动禁用渠道（AutoBan）
      • 判断是否应该重试（shouldRetry()）：
        • 重试：429, 307, 5xx（除超时）
        • 不重试：400, 408, 504, 524, 2xx, Skip Retry Error
      • 不重试 → 跳出循环
  • ⑧ 配额处理（最终）
    • 成功：在 DoResponse 中进行配额结算
    • 失败：退还预扣配额
7
适配器处理（Adaptor）
文件：relay/relay_adaptor.go + relay/channel/xxx/
  • ① 获取适配器
    • 根据渠道类型获取对应的适配器实例
    • adaptor := GetAdaptor(channelType)
    • 例如：OpenAI → openai.Adaptor, Claude → claude.Adaptor
  • ② 初始化适配器
    • adaptor.Init(relayInfo)
    • 设置渠道配置、API Key 等
  • ③ 请求转换
    ConvertRequest()
    • 模型映射：
      如果配置了 model_mapping
      将请求的模型名映射到上游模型名
      例如：gpt-4 → gpt-4-turbo
    • 参数覆盖：
      如果配置了 param_override
      强制覆盖请求参数
      例如：temperature → 0.7, max_tokens → 2000
    • 格式转换：
      将统一格式转换为上游 API 格式
      例如：OpenAI ↔ Claude ↔ Gemini
  • ④ 构建请求 URL
    • url := adaptor.GetRequestURL(relayInfo)
    • 如果配置了自定义 BaseURL → 使用自定义 URL
    • 否则 → 使用默认上游 API URL
  • ⑤ 设置请求头
    SetupRequestHeader()
    • 认证信息：
      • OpenAI：Authorization: Bearer xxx
      • Claude：x-api-key: xxx
      • Gemini：x-goog-api-key: xxx
    • Organization（如果有）：
      OpenAI-Organization: xxx
    • 请求头覆盖：
      如果配置了 header_override
      添加或覆盖请求头
  • ⑥ 发送请求
    • resp := adaptor.DoRequest(c, relayInfo, requestBody)
    • 发送 HTTP 请求到上游 API
    • 处理超时、连接错误等
8
上游 API 处理
OpenAI / Claude / Gemini / 阿里云 / ...
  • 上游 API 服务商接收请求
    • 验证 API Key
    • 验证请求格式
    • 处理请求
  • 生成响应
    • 调用 AI 模型生成内容
    • 计算 token 消耗（prompt_tokens, completion_tokens）
    • 构建响应（流式或非流式）
  • 返回响应给适配器
    • 流式：Server-Sent Events (SSE)
    • 非流式：JSON 响应
9
响应处理与配额结算
文件：relay/xxx/adaptor.go::DoResponse()
  • ① 响应处理
    流式响应（Stream）：
    • 逐块读取 SSE 数据
    • 解析每个 chunk
    • 提取内容和 usage 信息
    • 转发给客户端（实时）
    • 累计 token 消耗
    非流式响应（Non-Stream）：
    • 一次性读取完整响应
    • 解析 JSON
    • 提取内容和 usage 信息
  • ② 解析 usage
    • 提取 prompt_tokens
    • 提取 completion_tokens
    • 计算 total_tokens
  • ③ 计算实际配额消耗
    • 根据 prompt_tokens 和 completion_tokens 计算
    • 应用模型价格
    • 计算最终配额
  • ④ 配额结算
    PostConsumeQuota()
    • 扣除 Token 配额：
      DecreaseTokenQuota(tokenId, quota)
    • 扣除用户配额：
      DecreaseUserQuota(userId, quota)
    • 更新渠道已用配额：
      UpdateChannelUsedQuota(channelId, quota)
    • 退还预扣差额：
      如果实际消耗 < 预扣金额
      退还差额到 Token 和用户配额
  • ⑤ 记录日志
    • 记录请求日志（log 表）
    • 记录 token 消耗
    • 记录配额消耗
    • 记录响应时间
  • ⑥ 更新统计
    • 更新渠道响应时间
    • 更新渠道成功率
    • 更新用户使用统计
10
返回客户端
完成请求处理
  • ① 响应格式
    • 统一的 OpenAI 格式（或原始格式）
    • 包含生成的内容
    • 包含 usage 信息
  • ② 响应方式
    流式响应（stream=true）：
    • 逐块返回（Server-Sent Events）
    • 客户端实时接收
    • 最后一个块包含 usage
    非流式响应（stream=false）：
    • 一次性返回完整 JSON
    • 包含完整内容和 usage
  • ③ 客户端接收
    • 解析响应
    • 提取生成的内容
    • 查看 token 消耗
    • 完成调用
