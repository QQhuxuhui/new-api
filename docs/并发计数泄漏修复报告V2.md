# 并发计数泄漏问题彻底修复报告（V2）

## 问题回顾

线上 `claude_主力` 渠道显示有 3 个并发一直下不去，经排查发现是并发计数泄漏导致。

### 初次修复的遗留问题

第一版修复虽然解决了部分问题，但存在**三个严重缺陷**：

#### [高优先级] 问题 1: TTL 重置导致清理失效

**原因：**
- 每次 `Incr` 操作都重置 TTL 为 1 小时
- 泄漏的 key 保持高 TTL，永远不会被清理逻辑检测到
- 原清理逻辑只清理 TTL < 5 分钟的 key，实际上永远不会触发

**影响：** 泄漏的并发计数需要等待 1 小时才能被清理，**实质上等于没解决问题**。

#### [中优先级] 问题 2: 扫描可能不完整

**原因：**
- 单次 30 秒超时，key 多时会中断
- 下次扫描又从 cursor 0 重新开始
- 导致后面的 key 永远不会被检查

**影响：** 大量 key 时，清理任务失效。

#### [低优先级] 问题 3: 诊断脚本功能缺失

**原因：**
- `checkChannelConcurrency` 函数只是提示信息
- 承诺的"查询渠道"功能未实现

**影响：** 误导用户。

---

## 彻底修复方案（V2）

### 核心设计：基于时间戳的泄漏检测

采用**独立时间戳追踪**方案，不依赖 TTL：

```
并发计数 key: channel:key:{hash}:type:{type}:concurrent
时间戳 key:   channel:key:{hash}:type:{type}:concurrent:timestamp
```

**工作原理：**
1. 第一次 `Incr` 时，使用 `SetNX` 设置时间戳（仅首次设置）
2. 时间戳记录并发计数开始的时间
3. 清理任务检查时间戳，如果超过阈值（默认 2 分钟）则清理

**优势：**
- ✓ 不依赖 TTL，不会被重置影响
- ✓ 精确控制泄漏检测阈值（可配置）
- ✓ 兼容旧版本 key（自动补充时间戳）

## 详细修复内容

### 修复 1: 添加时间戳追踪机制

**文件：** `service/concurrency.go`

#### 新增函数
```go
func GetConcurrencyTimestampKey(concurrencyKey string) string {
    return concurrencyKey + ":timestamp"
}
```

#### 修改并发递增逻辑
```go
// Increment counter and set timestamp using pipeline
_, err = tx.TxPipelined(ctx, func(pipe redis.Pipeliner) error {
    pipe.Incr(ctx, redisKey)
    pipe.Expire(ctx, redisKey, 1*time.Hour)

    // Set timestamp only if it doesn't exist (first increment)
    // 使用 SetNX 确保只在第一次设置时间戳
    pipe.SetNX(ctx, timestampKey, time.Now().Unix(), 1*time.Hour)
    return nil
})
```

#### 修改并发递减逻辑
```go
// If counter goes to 0 or negative, clean up
if val <= 0 {
    // Delete both the counter and timestamp keys
    common.RDB.Del(ctx, redisKey, timestampKey)
}
```

### 修复 2: 重写清理逻辑

**文件：** `service/concurrency_cleanup.go`

#### 核心改进

1. **基于时间戳判断泄漏**
```go
// 检查是否超过泄漏阈值
age := time.Duration(now-timestamp) * time.Second
if age > ConcurrencyLeakThreshold {  // 默认 2 分钟
    // 删除泄漏的并发计数
    common.RDB.Del(ctx, key, timestampKey)
}
```

2. **移除整体超时限制**
```go
// 修改前：30 秒整体超时，可能导致扫描不完整
ctx, cancel := context.WithTimeout(context.Background(), 30*time.Second)

// 修改后：无整体超时，单次操作有合理超时
ctx := context.Background()
scanCtx, cancel := context.WithTimeout(ctx, 5*time.Second)
```

3. **支持环境变量配置阈值**
```go
// 默认 2 分钟，支持环境变量 CONCURRENCY_LEAK_THRESHOLD（秒）
var ConcurrencyLeakThreshold = 2 * time.Minute
```

4. **兼容旧版本 key**
```go
if err == redis.Nil {
    // 时间戳不存在，为旧版本 key 设置时间戳
    common.RDB.SetNX(ctx, timestampKey, now, 1*time.Hour)
    continue
}
```

5. **优化清理频率**
- 修改前：每 10 分钟执行一次
- 修改后：每 5 分钟执行一次（比泄漏阈值更频繁）

### 修复 3: 增强诊断脚本

**文件：** `scripts/diagnose_concurrency.go`

#### 新增三种模式

1. **概览模式（默认）**
```bash
go run scripts/diagnose_concurrency.go
```
显示：并发 key 数量、TTL

2. **详细模式**
```bash
go run scripts/diagnose_concurrency.go detail
```
显示：并发数、存在时长、泄漏警告（超过 2 分钟标记 ⚠️）

3. **清理模式**
```bash
go run scripts/diagnose_concurrency.go clear
```
清理所有并发计数和时间戳

#### 智能检测
```go
// 如果超过 2 分钟，标记为可能泄漏
if age > 2*time.Minute {
    leakWarning = " ⚠️  可能泄漏！"
}
```

## 配置说明

### 环境变量

```bash
# 设置泄漏检测阈值（单位：秒，默认 120）
```

### 推荐配置

| 场景 | 阈值设置 | 说明 |
|------|----------|------|
| 生产环境（默认） | 120 秒（2 分钟） | 平衡误判和及时清理 |
| 高并发环境 | 300 秒（5 分钟） | 避免正常长时间请求被误判 |
| 测试环境 | 60 秒（1 分钟） | 快速发现泄漏 |

## 性能影响

### Redis 操作增加

| 操作 | 修改前 | 修改后 | 增量 |
|------|--------|--------|------|
| 请求开始 | 1 次 (Incr) | 2 次 (Incr + SetNX) | +1 |
| 请求结束 | 1 次 (Decr) | 1-2 次 (Decr + Del) | +0~1 |
| 后台清理 | 2N 次 (Get + TTL) | 3N 次 (Get + Get timestamp + Del) | +N |

**影响评估：**
- 请求处理：增加 1-2 次 Redis 操作（SetNX 和可能的 Del）
- 由于使用 Pipeline，延迟几乎不变
- 清理任务：扫描更完整，但单次操作更多
- **总体影响：** 可忽略（< 1ms）

### 内存使用

- 每个并发计数增加一个时间戳 key（约 20 字节）
- 100 个并发 key ≈ 2KB 额外内存
- **影响：** 可忽略

## 故障排查

### 问题 1: 清理任务不生效

**可能原因：**
- Redis 未启用
- 清理任务未启动

**解决方案：**
```bash
# 检查日志
grep "Concurrency cleanup task started" logs/app.log

# 手动触发清理
go run scripts/cleanup_concurrency.go
```

### 问题 2: 误报泄漏

**可能原因：**
- 阈值设置过小
- 确实有慢请求

**解决方案：**
```bash
# 增加阈值（如 5 分钟）
export CONCURRENCY_LEAK_THRESHOLD=300
```

### 问题 3: 旧版本 key 清理

**现象：**
- 有 key 但无时间戳

**解决方案：**
自动处理，清理任务会自动为旧 key 设置时间戳
